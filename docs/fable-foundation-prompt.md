# Claude Fable 5向け：根幹一括実装プロンプト

以下を、Claude Codeでこのリポジトリを開いた状態のClaude Fable 5へ、そのまま1回で渡す。

---

あなたはこのリポジトリのリードエンジニアです。既存の文書だけの状態から、iOS／Android向け「OshiTrip」の根幹を、実行可能なFlutterアプリとして一括実装してください。

この依頼の目的は、全MVP機能を浅く並べることではありません。プロダクトの中心体験である次の一連の流れを、後から別モデルが安全に機能追加・修正できる品質で完成させることです。

```text
初回起動 → ログイン → 現場を登録 → 準備情報を管理
→ 当日に必要情報を即確認 → 終演後に思い出を記録
```

## 1. 最初に行うこと

1. `docs/requirements.md`、`docs/architecture.md`、`docs/adr/`、`docs/scaffold.md`、`docs/setup.md`をすべて読む。
2. 要件が競合する場合は、`requirements.md` → 承認済みADR → `architecture.md` → 本プロンプトの順で優先する。ただし、本プロンプトに明記した今回の実装範囲は守る。
3. 現在のファイルとGit差分を確認し、既存の`docs/`、`.github/`、`.gitignore`、`.env.example`を保持する。ユーザーの既存変更を破棄しない。
4. 実装前に短い作業計画を作り、そのまま実装を開始する。仕様確認の質問で停止しない。曖昧な点は、要件とUX原則から保守的に決めて`docs/decisions.md`へ記録する。
5. Flutter SDKや必要なCLIが利用できるか確認する。利用できる場合は生成・依存解決・コード生成・テスト・解析・ビルドまで実行する。利用できない場合もソース一式は完成させ、未検証項目と再現コマンドを明記する。成功していない検証を成功と報告しない。

## 2. 固定する技術構成

- Flutter / Dart、iOS 15以上、Android API 26以上
- 状態管理：Riverpod
- ルーティング：go_routerの`StatefulShellRoute`
- バックエンド：Supabase（Auth / Postgres / Storage）
- ローカルDB：Drift（SQLite）
- モデル：freezed + json_serializableを基本とする
- 日時：端末タイムゾーンを明示的に扱い、現在時刻は注入可能な`Clock`経由に限定する
- 構成：feature-first。presentation / application / domain / dataの責務を分離する
- domain層はFlutter、Riverpod、Supabase、Driftへ依存させない
- Repository抽象はdomain、実装はdataに置く
- 例外を画面まで素通しせず、型付き`Result` / `Failure`へ変換する
- development / staging / productionの3 flavorと、各環境用エントリポイントを作る
- UI文言は日本語を基準にし、テーマはライト／ダーク両対応とする

依存パッケージは、作業時点のFlutter stableと互換性が確認できる安定版を採用してlockファイルへ固定する。古いサンプルのバージョンを推測で固定しない。

## 3. 今回、実動まで完成させる範囲

### 3.1 アプリ基盤

- 共通`bootstrap`、環境設定、Supabase初期化、Riverpod、グローバルエラーハンドリング
- 未認証、チュートリアル未完了、認証済みを扱うルートガード
- 5タブ：ホーム／現場／思い出／マイ推し／設定
- 共通のloading / empty / error / offline表示
- アクセシブルなテーマ、十分なタップ領域、Semanticsラベル、色以外の状態表現
- センシティブ情報をマスクするロガー。チケット画像、座席、整理番号、予約番号、住所、交通・宿泊詳細を出力しない

### 3.2 初回起動と認証

- 最大4画面のスキップ可能なチュートリアル
- 完了状態の端末保存と、設定からの再表示
- Supabase Authによるメールアドレス＋パスワードの新規登録、ログイン、ログアウト
- パスワード再設定へ進める導線
- 通知権限は初回起動時に要求しない
- Supabase環境値が未設定のdevelopmentビルドでは、明示的に「デモモード」と表示したローカル利用を可能にする。本番でデモモードへ暗黙フォールバックしない

### 3.3 現場（本実装の中心集約）

- 最小必須項目：グループ／アーティスト名、公演名、日付
- 任意項目：会場、開場・開演・終演予定時刻、公演種別、推しメン、遠征有無、公演マスタとの関連
- 作成、一覧、詳細、編集、中止。削除は確認ダイアログ必須
- 登録フォームは必須項目から始め、追加情報を段階的に開示する
- 入力途中を自動保存し、再開可能にする
- 状態は「予定／準備中／本日／余韻中／思い出／中止」。日付・時刻と明示操作から純粋なドメインロジックで導出する
- 日跨ぎ、深夜公演、終演予定なし、日程変更を壊さない

現場詳細では、次を子データとして追加・編集・削除できるようにする。

- チケット：取得状況、支払状況、発券状況、座席、整理番号、入場口、URL、画像参照、メモ
- 交通：要否、交通手段、出発地／到着地、出発／到着時刻、往路／復路、予約番号、URL、メモ
- 宿泊：要否、宿泊先名、チェックイン／アウト日、住所、予約番号、URL、メモ
- Todo：名前、期限、完了状態、担当、重要度、メモ
- メモ：自由メモ、物販、集合場所、周辺施設、注意事項

「不要」と「未登録」を区別し、不要項目を未完了数やリマインド候補へ含めない。

### 3.4 ホームと当日ホーム

- 通常ホームは未来の現場を近い順に表示する
- 各カードに、公演名、推し、会場、日時、残日数、未完了Todo数、チケット／交通／宿泊の準備状態、次に行う1アクションを表示する
- 公演当日は最上部を当日モードに切り替え、チケット、会場、開場／開演／終演予定、座席／整理番号、集合場所、重要Todo、往復交通、宿泊を少ないタップで確認できるようにする
- 当日重要情報はDriftへキャッシュし、オフラインでも直近同期済み内容を表示する
- 外部チケットURLと保存画像を視覚的にもデータ上も区別する
- 終演予定後は「余韻中」と短い感想入力への導線を表示する

### 3.5 思い出

- 終了した現場は同一IDのまま「思い出」一覧へ表示区分を移す。コピーや別レコード化をしない
- 終演直後：写真参照、短い感想、特によかった点
- 翌日：感想本文、MC／当日メモ、セトリ、座席／見え方
- 後日：写真整理用情報、表情・タグ、グッズ／戦利品、周辺で行った場所／食べたもの、追記
- すべて任意入力。未入力をエラー扱いしない
- 一覧と詳細を実装し、現場情報と記録内容を一体で閲覧できるようにする

画像について、今回はローカル画像参照とアップロード用Repository境界まで実装する。認可付きStorageアップロードの完全な再試行UIは後続範囲でよいが、ドメインモデルを仮の文字列フィールドだけで済ませない。

### 3.6 マイ推しと設定の基礎

- グループ／アーティストとメンバーのローカルCRUD
- 推し区分：最推し／推し／ゆる推し／箱推し／気になる
- 色、推し始めた日、記念日、メモ
- 現場作成時に選択できる
- 設定：テーマ、チュートリアル再表示、アカウント情報、ログアウト、データ削除導線、アプリ情報
- アカウント削除は危険操作として再確認し、RepositoryとSupabase RPCの境界を用意する。未実装のサーバー処理を成功したように見せない

## 4. 今回は「境界と土台」までに留める範囲

次は画面や関数の大量な仮実装を作らない。後続モデルが追加できる、型・Repository境界・ルート予約・DB拡張余地・短いTODO文書までに留める。

- ユーザー投稿型公演DBの検索、重複統合、登録者数集計、通報、管理画面
- 他ユーザーとの共有招待、Realtime共同編集、項目単位の共有設定UI
- FCM/APNs登録、通知スケジューリング、通知設定詳細、ディープリンク実処理
- 写真のバックグラウンドアップロード、再試行キュー、サムネイル生成
- 課金、広告、アフィリエイト
- 分析イベント、クラッシュレポート、ストア提出自動化

`TODO`だけの空画面をタブや主要導線に表示しない。今回対象外の機能は、ユーザーが誤って利用可能だと思うUIを出さない。

## 5. データと同期

Supabase migrationを作り、最低限次の関係を正規化して表現する。

```text
profiles
oshi_groups -> oshi_members
performances
genbas -> tickets / transports / lodgings / todos / genba_memos
genbas -> memory_entries / memory_photos / setlist_items / goods_items / visited_places
outbox_operations
```

- ユーザー固有データには`owner_id`を持たせ、全対象テーブルでRLSを有効化する
- 所有者以外は既定で読み書き不可
- 公演マスタと個人の現場情報を混在させない
- `created_at`、`updated_at`、必要な外部キー、制約、索引を定義する
- 書き込みには`client_mutation_id`を持たせ、冪等性を確保できる形にする
- RLSの正負ケースをpgTAPでテストする
- ローカルDBは画面用キャッシュ、下書き、Outboxを扱う
- 書き込みはローカル反映 → Outbox → リモート同期とし、UIを通信完了まで止めない
- 同期状態をpending / syncing / synced / failed / conflictで表現し、失敗時に内容を失わない
- 最低限、現場作成・Todo更新・思い出入力でこの流れを実動させる
- 接続監視と再試行は抽象化し、テストではfakeへ差し替えられるようにする
- 競合解決は今回は`updated_at`によるlast-write-winsを既定とし、競合記録を残す。将来のフィールド単位マージを妨げない

## 6. UI・設計上の判断基準

- 派手な装飾より、遠征前と当日の速い確認を優先する
- 主要操作は片手で届く位置へ置く
- 最初から巨大フォームを見せない
- 空状態には説明と次の1アクションを置く
- 保存状態（保存中／端末保存済み／同期済み／同期失敗）を必要な場面だけ明示する
- 推しカラーはアクセントに限定し、文字コントラストを壊さない
- 画面からSupabaseやDriftを直接呼ばない
- 1ファイルへ複数機能を詰め込まない一方、意味のない薄いラッパーや過剰な抽象化も増やさない
- 将来機能のためだけの未使用コードを大量生成しない

## 7. テストと完了条件

最低限、次を自動テストする。

### domain / unit

- 現場状態遷移の日時境界、日跨ぎ、深夜公演、中止、日程変更
- 「不要」と「未登録」の集計差
- Result / Failure変換
- センシティブ情報のログマスキング
- Outboxの冪等再送、失敗保持、last-write-wins

### notifier / repository

- loading / empty / data / error / offline
- 現場作成、Todo完了、思い出入力の楽観更新とロールバック
- キャッシュ先行表示とバックグラウンド更新

### widget / routing

- 未認証／チュートリアル未完了のredirect
- 5タブの状態保持
- 現場作成 → ホーム表示 → 当日表示 → 思い出表示の主要フロー
- 主要操作のSemanticsラベル

### database

- 所有者は自分のデータをCRUDできる
- 他ユーザーは取得も更新もできない
- 子テーブル経由で所有権を迂回できない

完了時に、利用可能な環境の範囲で次を実行する。

```text
dart format --set-exit-if-changed .
flutter analyze
flutter test
flutter test integration_test
flutter build apk --debug --flavor development -t lib/main_development.dart
supabase test db
```

コード生成が必要なら先に実行する。テストを通すためにテストを削除・skip・弱体化しない。警告を大量に残したまま完了扱いにしない。

## 8. ドキュメントと最終報告

- `README.md`を、初見の開発者がdevelopmentビルドを起動できる内容へ更新する
- `docs/implementation-status.md`を、実際に確認できた内容だけで更新する
- 主要な設計判断と保留事項を`docs/decisions.md`へ記録する
- 後続モデル向けに`docs/follow-up-work.md`を作り、今回対象外の作業を依存順に分解する
- 既存の`docs/fable-post-implementation-review-prompt.md`と`docs/post-fable-phase-prompts.md`は、実装後の独立レビューと後続工程に使うため削除・上書きしない
- `.env.example`に必要なキー名だけを追加し、実値や秘密情報をコミットしない

最終回答では次だけを簡潔に報告する。

1. 実装した中心フロー
2. 主な設計判断
3. 実行した検証と、その成否
4. 環境不足で未検証の項目
5. 後続作業の優先順位

最後に、次の作業として新しい会話で`docs/fable-post-implementation-review-prompt.md`を実行すべきことを案内してください。レビューでCritical／Highが0件になった後は、`docs/post-fable-phase-prompts.md`のPhase順に進みます。

## 9. 禁止事項

- 既存文書やユーザー変更の無断削除
- 認可をクライアントUIだけで済ませること
- 本番環境でのデモデータやローカル認証への暗黙フォールバック
- APIキー、サービスロールキー、個人情報のコミットやログ出力
- 画面からSupabase／Driftを直接呼ぶこと
- すべてを1つの巨大ファイルへ実装すること
- コンパイルしないダミーコード、常に成功を返す未実装Repository、見せかけだけのテスト
- 対象外機能を中途半端なUIとして大量に並べること
- ツール未導入を理由に、実装自体を行わず計画書だけで終了すること

ここまでを一つの連続したタスクとして遂行してください。途中で小さな判断確認を求めて停止せず、安全な仮定を記録して前進してください。
