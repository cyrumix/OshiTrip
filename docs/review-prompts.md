# Phase別レビュー用プロンプト集

## 使い方

各Phaseの実装完了後、実装担当とは別の会話でレビューする。最初に「共通レビュープロンプト」を渡し、続けて対象Phaseのレビュープロンプトを渡す。

レビューでは原則としてコードを変更しない。指摘を確認した後、修正を別途依頼する。修正後は同じPhaseのレビューを再実行し、Critical／Highが0件になってから次Phaseへ進む。

## 共通レビュープロンプト

```text
あなたは、このプロジェクトの実装担当とは独立したシニアモバイルアーキテクト兼QA・セキュリティレビュアーです。現在指定されたPhaseの成果物を、仕様適合性、実装品質、テスト証拠、過不足の観点で厳格にレビューしてください。

最初に以下を全文読んでください。

- README.md
- docs/requirements.md
- docs/design-spec.md
- docs/implementation-prompts.md
- docs/architecture.md（存在する場合）
- docs/adr/配下（存在する場合）
- docs/implementation-status.md（存在する場合）
- docs/requirements-traceability.md（存在する場合）
- 対象Phaseのコード、DBマイグレーション、認可ポリシー、設定、テスト、CI
- 対象Phase開始前からの差分と未コミット変更

レビュー原則:

1. docs/requirements.mdを要件の唯一の正とし、実装プロンプトは担当範囲と完了条件の定義として扱う。
2. 実装プロンプトに書かれていなくても、仕様書上そのPhaseに必要な前提・例外・非機能要件を確認する。
3. コードが存在するだけで完了と判断しない。実際の画面導線、永続化、再起動後復元、サーバー側認可、エラー処理、自動テストを証拠として確認する。
4. モック、TODO、固定値、ダミーデータ、未接続UI、常に成功するスタブを検索し、実機能と誤認されていないか確認する。
5. 正常系だけでなく、空状態、通信失敗、低速通信、オフライン、再試行、二重タップ、二重送信、競合、タイムゾーン、日跨ぎ、深夜公演、日程変更、中止を対象にする。
6. チケット画像、座席、予約番号、交通、宿泊、個人感想、写真がログ、分析、クラッシュレポート、公開DB、未認可URLへ漏れないことを確認する。
7. UI上の非表示を認可とみなさず、API、DB、画像ストレージのサーバー側制約を確認する。
8. 色だけに依存しない状態表現、文字拡大、読み上げラベル、コントラスト、タップ領域、ダークモードを確認する。
8a. UI変更を含むPhaseではdesign-spec.mdを正とし、情報階層、Design Token、共通コンポーネント、写真あり／なし、推しカラー、ライト／ダークを確認する。ピクセル一致だけを合否基準にしない。
9. 対象Phaseより後の機能が未実装であること自体は問題にしない。ただし現在のPhaseに不要な先行実装、複雑化、仕様外機能は過剰実装として指摘する。
10. リポジトリへ変更を加えない。レビューのための読み取り、静的解析、テスト、ビルドは実行してよい。

必ず実施する確認:

- git差分、未追跡ファイル、依存変更を確認する。
- requirements-traceability.mdの対象行を仕様書と突合する。
- 静的解析、フォーマット確認、対象テスト、可能なビルドを実行する。
- テストが実装詳細の追認ではなく、要件上の振る舞いを検証しているか読む。
- テストを一時的に削除・無効化・スキップして通していないか確認する。
- 秘密情報、本番キー、個人データが含まれていないか確認する。

指摘の重大度:

- Critical: 情報漏えい、認証・認可回避、不可逆なデータ損失、主要導線が利用不能、ストア公開不能
- High: 必須要件の欠落、主要導線の不成立、誤った状態遷移、重要なテスト欠如、Phase完了条件未達
- Medium: 例外処理、UX、アクセシビリティ、保守性、テスト範囲の実質的な不足
- Low: 軽微な表示、命名、文書、将来の改善

出力形式:

1. 判定: PASS／CONDITIONAL PASS／FAIL
2. Phase完了条件: 条件ごとに「達成／一部達成／未達」と証拠
3. 要件トレーサビリティ: 対象要件ごとに、実装箇所、テスト、判定
4. 指摘一覧: 重大度順。各指摘に要件番号、ファイルと行、再現条件、影響、期待動作、修正方針を含める
5. 過剰実装・スコープ逸脱
6. 実行したコマンドと結果
7. 自動確認できなかった事項と理由
8. 残余リスク
9. 次Phaseへ進めるか: Yes／No。Noの場合は必須修正項目を列挙

根拠のない推測を事実として書かないでください。問題が見つからない場合も、確認した証拠と未確認範囲を明示してください。
```

## Phase 0レビュー: 技術設計と土台

```text
Phase 0をレビューしてください。共通レビュープロンプトに加え、以下を重点確認してください。

仕様・設計:

- Flutter／DartでiOS・Androidの両方が正しく初期化されている。
- Windowsでの日常開発とmacOS CIでのiOS検証の境界が明確である。
- バックエンド選定が、認証、RDB、画像、リアルタイム共有、Push通知、運用性、費用、ロックインを比較したADRに基づく。
- 状態管理、ルーティング、Repository、API、ローカルキャッシュ、エラー表現の責務が分離されている。
- development／staging／productionが混線せず、秘密情報をコードへ含めない。
- iOS／Androidの最低OSバージョンが根拠付きで決定されている。

データ保護:

- DBマイグレーション、所有者認可、行単位アクセス制御、画像認可をテストできる土台がある。
- 管理者権限が通常ユーザーへ露出しない設計である。
- 端末キャッシュとログのデータ保護方針がある。

品質基盤:

- lint、format、単体テスト、Widgetテスト、CIが実際に動作する。
- WindowsでAndroidデバッグビルドが成功する。
- macOS CIで署名不要のiOSビルドとテストが実際に成功する。設計書や手順だけの場合は未達とする。
- requirements-traceability.mdで仕様書の全節がPhaseまたは非MVPバックログへ割り当てられ、未割当がない。
- 不要な製品機能を先行実装していない。

追加で、依存パッケージの保守状況、ライセンス、最低OSへの影響、既知脆弱性も確認してください。
```

## Phase 1レビュー: アプリ基盤・認証・初回体験

```text
Phase 1をレビューしてください。共通レビュープロンプトに加え、以下を重点確認してください。

認証:

- 新規登録、ログイン、セッション復元、期限切れ、ログアウト、認証失敗が動作する。
- ログアウト後に保護画面へ戻れず、保護キャッシュや認証情報が適切に削除される。
- 認証状態の競合や起動時フラッシュで保護情報が表示されない。

チュートリアル:

- 初回のみ最大4画面で表示され、スキップできる。
- 準備・当日・思い出の価値と「最初の現場を登録する」導線がある。
- チュートリアル中にOS通知権限を要求しない。
- 完了・スキップ状態が再起動後も維持される。
- 設定から何度でも再表示でき、再表示しても初回状態を壊さない。

ナビゲーション・設定:

- ホーム、現場、思い出、マイ推し、設定の5タブがあり、戻る操作とタブ状態保持が自然である。
- アカウント、テーマ、推しカラー、ダークモード、タイムゾーン設定が保存・復元される。
- ローディング、空状態、通信エラー、再試行が共通UIとして機能する。
- 「現場登録」導線は存在するが、Phase 2の機能をダミーで完成扱いにしていない。

アクセシビリティ:

- 文字拡大、読み上げ、色以外の識別、タップ領域、ダークモードをWidget／Golden／手動確認の適切な組み合わせで検証している。
```

## Phase 2レビュー: 現場登録・準備管理

```text
Phase 2をレビューしてください。共通レビュープロンプトに加え、以下を重点確認してください。

現場登録:

- グループ／アーティスト、公演名、日付の3項目だけで保存できる。
- 会場、開場・開演・終演時刻、公演種別、推しメン、遠征有無を後から追加できる。
- 初回登録後に残り日数、おすすめTodo、次の準備アクションが表示される。
- 入力途中の自動保存、中断再開、検証、削除確認が動作する。

準備情報:

- チケット、交通、宿泊、Todo、メモについて、仕様書記載の全項目が保存・編集・削除・復元できる。
- 交通・宿泊の「未登録」と「不要／なし」がデータ上もUI上も異なる。
- 「不要／なし」を選ぶと未準備表示、通知候補、アフィリエイト候補から除外される。
- Todoテンプレートが現場種別と遠征有無に対応し、適用前確認と一括解除ができる。
- ホームカードに必要情報、残り日数、未完了Todo、準備状態、次のアクションが表示される。

状態・日時:

- 予定、準備中、本日、余韻中、思い出、中止の状態判定が仕様と一致する。
- タイムゾーン、日跨ぎ、深夜公演、終演時刻未入力、複数公演、日程変更、中止をテストしている。

永続化・認可:

- 現場、チケット、交通、宿泊、Todo、メモのマイグレーションと外部キー・制約がある。
- 他ユーザーが非共有データをID推測、API直接呼び出し、DBアクセスで取得・変更できない。
- 再送や二重タップで重複データが作られず、競合方針が実装されている。
- センシティブ値がログへ出ない。
```

## Phase 3レビュー: 当日ホーム・オフライン

```text
Phase 3をレビューしてください。共通レビュープロンプトに加え、以下を重点確認してください。

当日ホーム:

- 当日の現場が通常カードより優先され、ホーム最上部へ表示される。
- チケット、会場・地図・入場口、開場・開演・終演、座席・整理番号、集合場所、重要Todo、往復交通、宿泊へ1〜2操作で到達できる。
- 複数の当日公演が開始時刻順に表示される。
- 終演予定時刻後に余韻中へ切り替わり、写真と短い感想の導線が出る。
- 外部チケットリンクと保存画像が誤認されない。

オフライン:

- 通信を切った実テストで、直近同期済みの当日重要情報を閲覧できる。
- 最終同期時刻、古い可能性、未取得状態が明確に表示される。
- 外部アプリ起動失敗、URL不正、キャッシュなし、同期失敗から回復できる。
- ローカルキャッシュが必要最小限で、保護対象は暗号化される。
- ログアウトやアカウント削除後に保護キャッシュを閲覧できない。

日時:

- 端末タイムゾーン変更、深夜終演、終演時刻未入力、日跨ぎ、複数公演について当日／余韻表示が正しい。

UX:

- 片手操作、文字拡大、屋外での視認性、読み上げ、主要ボタンのタップ領域を確認する。
- アフィリエイトや後続Phaseの機能が当日主要操作を妨げていない。
```

## Phase 4レビュー: 思い出と段階入力

```text
Phase 4をレビューしてください。共通レビュープロンプトに加え、以下を重点確認してください。

移行:

- 終了した現場が削除・複製されず、同一IDと登録情報を維持したまま思い出へ表示区分変更される。
- 過去の現場は通常ホーム・未来の現場一覧から外れ、思い出一覧に表示される。
- 日付訂正時に表示区分が再判定され、データが失われない。

段階入力:

- 終演直後は写真、短い感想、特によかった曲を優先する。
- 翌日は感想本文、MC・当日メモ、セトリ、座席・見え方を案内する。
- 後日は写真整理、表紙、タグ、グッズ、周辺の場所・食事、追記を案内する。
- ユーザーはどの時点でも翌日・後日の項目を先行入力できる。
- 短い感想が別データとして分断されず、感想本文の初期値として加筆できる。
- 空欄を未完成として責めるUIや、不必要な必須入力がない。

詳細機能:

- 写真、表紙、タグ、感想、セトリ、曲メモ、アンコール、特によかった曲、MC・当日メモ、座席・見え方、グッズ・戦利品の仕様項目が揃う。
- 感想がユーザー単位で保存されるデータ構造になっている。
- 自動保存と再起動後復元が動作する。

写真・セキュリティ:

- 非同期アップロード、進捗、失敗時再試行、二重登録防止がある。
- 写真失敗時に文章や他項目が失われない。
- 未認可ユーザーが画像URLやIDを知っていても取得できない。
- EXIFやログなど意図しない個人情報の扱いが検討・記録されている。

このPhaseではPush通知本体がPhase 6であることを考慮し、通知未実装だけを欠陥にしない。ただし通知判定に必要な入力状態が保存されているか確認してください。
```

## Phase 5レビュー: マイ推し・ユーザー投稿型公演DB

```text
Phase 5をレビューしてください。共通レビュープロンプトに加え、以下を重点確認してください。

マイ推し:

- グループ／アーティスト名、種別、カラー、メモを登録できる。
- メンバー名、所属、推し区分、カラー、推し始めた日、誕生日、メモ、画像を登録できる。
- 最推し、推し、ゆる推し、箱推し、気になるを扱える。
- 現場登録中に推しを同時作成でき、事前登録を強制しない。
- 現場が思い出になっても推しとの紐づけが維持される。

公演DB:

- 既存公演の検索・選択・自動入力と、新規追加の両方が動作する。
- インクリメンタル検索、表記揺れ、空結果、通信失敗を扱う。
- グループ、日付、会場、開演時間で重複候補を登録前に表示する。
- 候補順が完全一致、登録者数、日付の近さを基本にする。
- 表記が「登録者 ○人」で、参加・出席を保証しない説明がある。

公開・非公開境界:

- 公演名、会場、日時などの共通情報と、チケット、座席、交通、宿泊、Todo、メモなどの個人情報が別テーブル・別API境界である。
- 公演マスター入力経由で個人情報を共通領域へ混入できない。
- 人数集計を無効にしたユーザーは登録者数へ含まれない。
- 少人数時の具体人数非表示がADRの閾値どおり機能する。
- 集計値から個人を列挙・推測するAPIが存在しない。

運用:

- 変更履歴、通報、管理者修正の基盤がある。
- 複数ユーザー利用中の公演を作成者が変更した場合の影響と監査が定義されている。
- 管理者機能を通常ユーザーが実行できない。
```

## Phase 6レビュー: 共有・Push通知

```text
Phase 6をレビューしてください。共通レビュープロンプトに加え、以下を重点確認してください。

共有:

- 現場と思い出の招待、参加、拒否、解除、メンバー削除、権限変更、オーナー移譲が動作する。
- オーナー、編集可、閲覧のみについて、仕様書どおりの許可・拒否がサーバー側で保証される。
- チケット画像、予約番号、住所、個人感想を項目単位で非共有にでき、初期値が安全側である。
- 感想は共同編集されず、ユーザーごとに保持される。
- 共有解除後のキャッシュ、期限付き画像URL、Push通知から情報が残存しない。
- 同時編集、競合、削除中、権限変更中の操作が安全に処理される。
- 招待や権限変更が監査可能である。

通知:

- OS通知許可前に価値を説明し、拒否後も主要機能を利用できる。
- 現場前、Todo、必要な交通・宿泊、終演後、翌日、後日、記念日、共有更新を扱う。
- 「不要／なし」、完了済み、「今回は入力しない」は対象外になる。
- 終演後は写真または短い感想を案内する。
- 翌日は翌日対象項目がすべて未入力の場合だけ送る。
- 3〜7日後は後日対象項目がすべて未入力の場合に1回だけ送る。
- 同一現場・同一種別の通知が重複せず、送信判定が冪等である。
- 同日通知が適切に集約される。
- タイムゾーン変更、日程変更、中止、入力直後との競合で誤通知しない。
- 通知タップで認証状態を考慮し、正しい対象画面へ遷移する。
- 通知本文に座席、予約番号などのセンシティブ情報を出さない。

設定:

- 通知種類別の有効／無効とタイミングを変更できる。
- 共有範囲、検索可否、登録者数集計への参加、項目別公開範囲が保存・反映される。

Pushの実送信を自動化できない環境では、サーバー判定テストとstaging端末での送受信証拠を分けて評価し、未確認事項を明記してください。
```

## Phase 7レビュー: マネタイズ・品質・リリース準備

```text
Phase 7をレビューしてください。これはMVP全体の最終レビューでもあります。共通レビュープロンプトに加え、以下を重点確認してください。

マネタイズ:

- 宿泊が必要かつ未登録の場合だけ宿泊導線が表示される。
- 交通が必要かつ未登録の場合だけ交通導線が表示される。
- Todoや持ち物に関連する場合だけ商品導線が表示される。
- 「不要／なし」、登録済み、完了済みでは表示されない。
- アフィリエイトであることが明示され、当日ホームの主要操作より優先されない。
- 外部遷移失敗や不正URLを安全に処理する。

設定・データ主体の権利:

- アカウント、通知、表示、テーマ、推しカラー、ダークモード、タイムゾーン、プライバシー、人数集計、チュートリアル、データ出力、データ削除が一つの設定体験として完成している。
- データ出力が本人確認を伴い、現場、思い出、設定、画像一覧を機械可読形式で取得できる。
- 出力URLが期限付きで、他ユーザーが取得できない。
- アカウント削除が共有データとの関係を安全に処理し、保護キャッシュ、画像、トークンも削除する。

セキュリティ・プライバシー:

- 通信と保存データの暗号化、画像認可、サーバー側権限、ログ除外を確認する。
- ログ、クラッシュレポート、分析イベントにチケット、座席、予約番号、住所、感想、画像URLが含まれない。
- 実運用の秘密情報がリポジトリやアプリバイナリへ不用意に含まれない。
- 脆弱な依存、デバッグ設定、テスト用バックドアが残っていない。

品質:
```

## 修正後の再レビュー用プロンプト

```text
前回のPhaseレビュー指摘に対する修正を再レビューしてください。

最初に前回のレビュー結果と、修正後差分を確認してください。そのうえで:

1. 各指摘を「解消／一部解消／未解消／別問題が発生」に分類する。
2. 修正コードだけでなく、回帰テストと実行結果を確認する。
3. 修正により要件、認可、データ移行、既存機能へ新しい問題が生じていないか確認する。
4. 前回未確認だった事項も、環境が利用可能になった場合は確認する。
5. 新規指摘を前回分と分けて重大度順に示す。
6. Phase判定と次Phaseへ進めるかを更新する。

レビューのみ行い、コードは変更しないでください。
```
