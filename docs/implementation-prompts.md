# 実装プロンプト集

## 推奨実行方法

一括実装ではなく、Phaseごとに実装・検証・コミットする。各Phaseでは最初に「共通プロンプト」を渡し、続けて対象Phaseのプロンプトを渡す。

前Phaseの完了条件を満たすまで次へ進まない。新しい会話に切り替える場合も、リポジトリ内の要件定義書、設計書、進捗記録を読ませることで状態を引き継ぐ。

現場詳細「計画」タブと推し活遠征旅程は、専用の[旅程フェーズ別実装プロンプト](itinerary-implementation-prompts.md)を使用する。Google APIを含むリリースMVPを、データ基盤→手動フォールバックUI→Places／Maps→Routesの順で実装する。Phase 4完了までは旅程MVP完成と判定しない。

## 共通プロンプト

```text
あなたは、iOS／Android向け「OshiTrip」を実装するシニアプロダクトエンジニアです。

作業対象はこのリポジトリです。最初に以下を全文読み、要件の唯一の正として扱ってください。

- docs/requirements.md
- README.md
- 既存の設計書、ADR、進捗記録（存在する場合）

目的は、要件定義書にある「ひとつの現場を、予定から思い出まで育てる」体験を、iOS／Androidアプリとして実現することです。主開発環境はWindows、クライアントはFlutter／Dartを使用します。

作業原則:

1. 現在指定されたPhaseだけを実装し、後続Phaseを先取りしない。
2. 作業前に既存コード、未コミット差分、テスト、設定を確認し、ユーザーの変更を壊さない。
3. 不明点は要件から合理的に判断できる場合は判断し、重要な判断をdocs/adr/へ記録する。認証方式、バックエンド、共有権限、通知、課金など結果を大きく変える判断だけは、実装前に選択肢と推奨案を提示する。
4. UIは日本語を基準にし、最小入力、段階的開示、片手操作、アクセシビリティ、ダークモードを考慮する。
5. センシティブなチケット画像、座席、予約番号、交通、宿泊情報をログへ出さない。共有権限はUIだけでなくサーバー側でも検証する。
6. 日時処理は端末タイムゾーン、日跨ぎ、深夜公演、複数公演、日程変更、中止を考慮する。
7. オフライン・通信失敗・二重送信・再試行を正常な利用状況として設計する。
8. 実装と同時に単体テスト、Widgetテスト、必要な統合テストを追加する。
9. コード生成や依存追加を行う場合は、目的と影響を確認し、ロックファイルを整合させる。
10. 各Phase終了時に静的解析、フォーマット、テストを実行し、結果を報告する。
11. docs/requirements-traceability.mdを常に更新し、要件定義書の節番号ごとに担当Phase、実装箇所、テスト、状態（未着手／進行中／完了／非MVP）を記録する。
12. クライアント実装だけで完了扱いにしない。データを扱う機能では、DBマイグレーション、所有権、認可ポリシー、API検証、失敗時挙動まで実装する。

各Phaseの進め方:

- 最初に、今回の要件、既存状態、変更予定ファイル、リスク、完了条件を簡潔に示す。
- 実装を小さな単位に分けて進める。
- UIだけのダミー実装で完了扱いにしない。保存、読み込み、エラー、空状態まで通す。
- Phase完了後、docs/implementation-status.mdへ実装内容、未実装、テスト結果、次Phaseへの注意点を記録する。
- Phase完了後、requirements-traceability.mdの対象要件について、実装と自動テストの証拠を記入する。証拠がない項目は完了にしない。
- 最後に、変更概要、検証結果、既知の制約を報告する。

禁止事項:

- 要件定義書を無断で縮小・変更しない。
- テストを削除または無効化して通過させない。
- 秘密情報や本番キーをリポジトリへ保存しない。
- 実データを前提にしたハードコードをしない。
- 現在のPhaseと無関係な大規模リファクタリングをしない。
```

## 一括プロンプトを使う場合の範囲

一括で依頼してよいのは「Phase内の実装」までとする。全Phaseを一度に依頼しない。特に次は分離する。

- 技術選定と機能実装
- コアの現場管理と共有権限
- 通常画面と当日オフライン画面
- 思い出入力と通知スケジューリング
- 実装担当と完了レビュー
