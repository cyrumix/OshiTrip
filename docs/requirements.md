# OshiTrip 要件定義書

版: 2.1  
対象: iOS／Android  
主開発環境: Windows

## 1. プロダクト概要

本アプリは、推し活における「現場」の予定管理、遠征準備、当日確認、終了後の思い出記録を一体化するモバイルアプリである。

プロダクトの中心単位を「1現場」とし、次の時間軸を支える。

- 未来: 現場、チケット、交通、宿泊、Todoを管理する
- 当日: 必要な情報をホームからすぐ確認できる
- 過去: 終了した現場を思い出として保存・共有する
- 継続: 推しごとの現場履歴と思い出が蓄積される

コアコンセプトは「推しと過ごす一つひとつの現場を、準備から思い出まで残せるアプリ」とする。

## 2. UX原則

### 2.1 最小入力と段階的開示

最初からすべての入力欄を表示せず、その時点で必要な項目を優先表示する。ユーザーは任意で後続項目を先行入力できる。

- 現場登録時は必要最小限の項目で登録可能にする
- 遠征しないユーザーには交通・宿泊入力を強制しない
- チケット、交通、宿泊は状態に応じた項目だけを表示する
- 任意項目は「あとで入力」できる
- 入力内容は自動保存し、中断後に再開できる

### 2.2 現場を育てる体験

記録の完成を要求するのではなく、予定から準備、当日、思い出へと情報が自然に育つ体験を提供する。

### 2.3 当日の即時性

当日は移動中、片手操作、時間的な焦り、通信不良を想定し、必要情報をホームの最上部に集約する。

### 2.4 非公開を基本とする

推し活情報、座席、チケット、移動、宿泊はセンシティブな情報である。デフォルトは非公開とし、共有は明示操作でのみ有効にする。

### 2.5 写真と余韻を中心にした視覚体験

管理表のように文字だけを並べず、公演写真、思い出写真、推しカラー、残日数を活用して、現場前の期待と終了後の余韻が伝わる体験にする。ただし、写真未登録時もすべての主要機能が成立し、当日の可読性と操作速度を装飾より優先する。

詳細な画面構成、配色、共通コンポーネントは[UI・ビジュアルデザイン仕様書](design-spec.md)を正とする。

## 3. 対象ユーザー

- アイドル、アーティスト、声優、俳優、VTuber、舞台、2.5次元、K-POPなどの現場へ行く人
- 交通、宿泊、チケット、持ち物の管理が煩雑になる人
- 過去の現場を写真、セトリ、感想、メモとして残したい人
- 複数の推しやグループを管理したい人
- 同行者と現場情報や思い出を共有したい人

## 4. 初回起動・チュートリアル

### 4.1 初回チュートリアル

初回起動時にチュートリアルを表示する。目的は機能説明ではなく、最初の現場登録へ迷わず到達させることである。

チュートリアルは最大4画面とする。

1. 現場をひとつにまとめる価値を説明
2. 準備・当日・思い出の流れを説明
3. 通知とプライバシーの概要を説明
4. 「最初の現場を登録する」を主ボタンとして表示

仕様:

- 各画面にスキップを表示する
- 最後まで閲覧しなくても利用開始できる
- OS通知許可はチュートリアル中に要求せず、通知価値が発生した時点で説明後に要求する
- チュートリアル完了状態をアカウントまたは端末に保存する
- 設定の「チュートリアルをもう一度見る」から再表示できる

### 4.2 初回現場登録

初回登録時の必須項目は次のとおりとする。

- グループ／アーティスト
- 公演名
- 日付

会場と開演時間は登録を推奨するが、初回登録を妨げないため任意とする。既存の公演候補を選んだ場合は公演情報を自動入力する。

登録完了直後に次を表示し、初回価値を提供する。

- 公演までの残り日数
- 公演種別に応じたおすすめTodo
- 次に行うべき準備1件
- 「交通・宿泊を追加」「チケットを追加」「あとで」の選択肢

## 5. 情報設計とグローバルメニュー

主要メニューは次の5つとする。

```text
ホーム
現場
思い出
マイ推し
設定
```

- 「現場」には当日を含む未来の現場を表示する
- 「思い出」には終了した過去の現場を表示する
- 終了した現場が思い出へ移ることは、削除や別データへの変換ではなく表示区分の変更として扱う
- **終演済み（手動で終了、または終演予定時刻を過ぎた「余韻中」）にした瞬間から、その現場は「思い出」にも表示する。ただし公演当日のうちは「ホーム」「現場」一覧にも引き続き残す（当日は両方に出る）。翌日（終演日の翌0時以降）からは「思い出」のみに表示する。** 当日中に余韻・感想入力へすぐ移れる導線と、当日ホームでの最終確認の両立を意図する。

## 6. ホーム

### 6.1 通常時

未来の現場を日付が近い順に表示し、「次に何をすべきか」を即座に把握できるようにする。

現場カードの主要表示項目:

- グループ／アーティスト名
- 公演名
- 会場
- 日付・開演時間
- 公演までの日数
- 未完了Todo数
- チケット、交通、宿泊の準備状態
- 次に行うべきアクション

カード全体を警告表示にせず、未準備項目ごとに重要度を付ける。

### 6.2 当日ホーム

公演当日はホーム最上部を「本日の現場」モードに切り替える。通常の現場カードより大きく表示し、以下へ1タップでアクセス可能にする。

- チケットを表示
- 会場、地図、入場口を確認
- 開場・開演・終演予定時刻を確認
- 座席または整理番号を確認
- 集合場所を確認
- 未完了の重要Todoを確認
- 往路・復路の交通を確認
- 宿泊先を確認

当日の重要情報は可能な範囲で端末にキャッシュし、通信不良時も直近同期済み情報を閲覧可能にする。チケットサービスへの外部リンクと、保存済みチケット画像は明確に区別する。

終演予定時刻後は同じ領域を「余韻中」に切り替え、写真と短い感想の入力を案内する。

複数の当日公演がある場合は開始時刻順に表示する。

## 7. 現場

### 7.1 状態

現場は次の状態を持つ。

```text
予定
準備中
本日
余韻中
思い出
中止
```

状態は日付・時刻により自動判定するが、ユーザーが日程変更、中止、終了時刻を修正できる。日付が過去になった現場は「現場」一覧から外れ、「思い出」に表示する。**「余韻中」（終演済みだが当日中）は「思い出」に表示しつつ「現場」一覧・ホームにも残す二重表示とし、翌日の「思い出」へ切り替わる。**

**予想終演時間と実際の終演時間**: 最初に登録する終演時間は予想値である。公演後、ユーザーが余韻・思い出入力で「実際の終演時間」を記録できる。実際の終演時間を入力したら、確認のうえ現場の終演時間を上書きし、概要・当日ホーム・計画（終演アンカー）など終演時間を参照する箇所すべてに反映する。深夜終演（開演より前の時刻）は日跨ぎとして扱う。

**準備サマリの表示位置**: 現場詳細では、ヒーローカード → 準備サマリ → タブ（概要／Todo・持ち物／チケット／交通／宿泊／計画／メモ）の順に構成する。準備サマリはヒーローカードとタブの間に常時表示し、どのタブを開いていても準備状況（Todo・持ち物・チケット・交通・宿泊）を確認できるようにする。

### 7.2 現場作成

最小登録項目:

- グループ／アーティスト
- 公演名
- 日付

追加項目:

- 会場
- 開場・開演・終演予定時刻
- 公演種別（選択式。ライブ・コンサート／フェス／リリースイベント／特典会・撮影会／ファンミーティング／トークイベント／舞台・ミュージカル／展示会／スポーツ観戦／オンライン配信／その他。内部値は安定コード、表示は日本語ラベル。既存自由入力は移行し、変換不能は「その他」として元文字列を保持）
- 推しメン
- 遠征の有無
- 公演候補との紐づけ
- ユーザーが利用権限を持つ公演ヒーロー画像（チケット画像とは別管理）
- 参加状態（予定／参戦済み／不参加／中止）

現場作成後に詳細項目を段階的に追加できる。

### 7.3 チケット

- 取得状況
- 当落状況
- 発券状況
- 座席
- 整理番号
- 入場口
- チケットURL
- チケット画像
- メモ

チケット画像にはセンシティブ情報が含まれる可能性を表示し、共有時はデフォルトで共有対象外とする。

### 7.4 交通

- 交通手段（選択式。新幹線／在来線・電車／飛行機／高速バス・夜行バス／路線バス／自家用車／レンタカー／フェリー／タクシー／徒歩・自転車／その他。内部値は安定コード、表示は日本語ラベル。既存自由入力は移行し、変換不能は「その他」として元文字列を保持）
- 出発地・到着地
- 出発・到着時刻
- 往路・復路
- 予約番号
- 予約URL
- メモ

「遠征なし」または「交通の登録は不要」を選択できる。未登録と不要を区別し、不要な通知を送らない。

新規登録時、出発・到着時刻の初期日付は現場の開催日とする（往路・復路とも）。往路を新規保存すると「同様の経路で復路を登録しますか？」を確認し、承諾時は交通手段を引き継ぎ、出発地・到着地を逆にし、出発・到着時刻は未設定の復路を追加登録する。既存交通の編集時はこの確認を出さない。

### 7.5 宿泊

- 宿泊の要否
- 宿泊先名
- チェックイン・チェックアウト日
- 住所
- 予約番号
- 予約URL
- メモ

「宿泊なし」を選択できる。宿泊なしの場合は未登録通知と予約導線を表示しない。

新規登録時、チェックイン・チェックアウト日の初期値は現場の開催日とする。

### 7.6 Todo

- Todo名
- 期限
- 完了状態
- 担当者
- 重要度
- メモ
- テンプレート種別

現場種別、遠征有無、日数に応じてテンプレートを提案する。テンプレート適用前に追加内容を確認でき、不要項目は一括解除できる。

### 7.7 メモ

現場ごとに**複数のメモ**を作成・編集・削除・並び替えできる。各メモはタイトル・本文・
種類・並び順・作成日時・更新日時を持ち、保存・同期・削除はメモID単位で行う。
同じ種類のメモを複数作成できる。タイトルと本文がどちらも空のメモは保存しない。

作成時に選べるテンプレート（種類。入力欄の初期値・入力例を補助するもので、
テンプレート自体は保存データとして強制しない）:

- 自由メモ
- 物販
- 集合場所
- 周辺施設
- 注意事項
- テンプレートなし（自由に追加）

既存メモは削除せず新構造へ移行し、タイトルの初期値は種類名にする。オフライン編集と
Outbox 同期を維持する。

### 7.8 共有

権限はオーナー、編集可、閲覧のみとする。チケット画像、予約番号、住所、個人の感想は項目単位で共有可否を設定できるようにする。

#### 7.8.1 共有メンバーの追加方法（2種類）

現場共有をLINE/DM経由でもアプリ内でも自然に行えるよう、共有メンバーの追加を2方式用意する。

- **A. 招待URLを送る**: オーナーが現場ごとに招待URLを発行し、LINE/DM等で送る。相手はURLから参加確認画面へ遷移し、参加するとその現場の共有メンバー（`genba_shares`）に追加される。
  - 招待URLの発行・無効化はオーナーのみ。招待は現場単位。
  - 参加後の初期権限はオーナーが viewer / editor から選べる（既定は viewer）。
  - 招待URLは無効化でき、任意で有効期限・使用回数上限を設定できる。
  - 参加済みユーザーが同じURLを開いても重複追加せず、現場詳細へ遷移する。
  - 参加確認画面に表示: 現場名・公演名・日付・オーナーの表示名/アイコン・参加後の権限・「参加する」「キャンセル」。
  - トークンを知らない人は参加できない。無効化済み・期限切れ・上限超過の招待は使えない（サーバーで強制）。
- **B. フレンドから追加**: アプリ内のフレンド一覧から相手を選び、権限（viewer/editor）を指定して共有メンバーに追加する。

#### 7.8.2 簡易プロフィール

ユーザーは簡易プロフィール（表示名・アイコン画像[任意]・ひとこと[任意]・推し名/推しカテゴリ[任意]・フレンド申請を受け付けるか・検索可能にするか[任意]）を設定できる。最低限は表示名とアイコンを優先する。プロフィールは共有メンバー一覧・招待参加画面・フレンド一覧で表示する。プロフィールは本人のみ編集でき、可視範囲は本人・承認済みフレンド・同一現場の共有メンバー・検索許可(searchable)ユーザーに限る（フレンドでない相手を無制限に列挙・検索できない）。

各アカウントは**一意なフレンドコード**（サーバー採番・例 `OSHI-7K3P-Q9A2`・NOT NULL/UNIQUE・形式 CHECK 制約）を持つ。コードは推測しにくく読みやすい固定形式（曖昧文字 0/1/I/O を除外）。**フレンドコードはサーバー採番専用で、本人でも任意に変更・設定できない**（列レベル権限で `friend_code` 列を authenticated から書けなくし、加えて BEFORE UPDATE トリガで採番済み値の変更を拒否する）。新規プロフィール作成時に `friend_code` が null ならサーバーが自動採番し、既存値は保持する。自分のコードはプロフィール/フレンド画面で確認・コピーでき、相手に伝えるとフレンド申請の**明示的な到達手段**になる（§7.8.3）。

#### 7.8.3 フレンド機能（共同編集の補助）

ユーザー同士がフレンドになれる。状態は none/pending/accepted/rejected/(blocked 任意)。操作は申請送信・承認・拒否・削除・一覧表示・フレンドを現場共有メンバーへ追加。フレンド機能はSNS化させず共同編集の補助に留める。フレンド申請は次のいずれかの**明示的な到達手段**でのみ送れる（無制限検索は行わない）:

- **フレンドコード**（`send_friend_request_by_code`）: 相手のフレンドコードを**完全一致**で入力したときに申請できる。コードを知っていることを明示的な到達手段とみなすため、相手が `searchable=false` でも申請できる。部分一致・列挙はできない（＝無制限検索にはならない）。自分自身のコード・存在しないコードは拒否する。
- **同一現場の共有メンバー**（招待URLで同じ現場に参加した相手）または **`searchable=true`** のユーザー（`send_friend_request`。従来どおり）。

いずれの経路でも、受付可否（`accepts_friend_requests`）・ブロック・状態機械（相互申請の自然化・rejected からの再申請）はサーバー（SECURITY DEFINER RPC）で強制する。フレンド申請・応答は本人（requester/receiver）のみが操作できる。フレンド画面には自分のフレンドコード表示（コピー可）と「フレンドコードで追加」導線を置く。

#### 7.8.4 招待参加後のフレンド申請

招待URLで同じ現場に参加したユーザー同士は、共有メンバー一覧からフレンド申請できる。メンバー一覧にプロフィールを表示し、フレンドでない相手には「フレンド申請」、申請中は「申請中」、フレンド済みは「フレンド」を表示する。自分自身には申請ボタンを出さない。

#### 7.8.5 Deep Link

招待URLの最終形は `https://oshitrip.app/invite/{token}` とし、開くとアプリ内の参加確認画面へ遷移する。ネイティブは iOS Universal Links（Associated Domains・AASA）／Android App Links（intent-filter `autoVerify`・assetlinks.json）で受け、go_router が受信パス `/invite/{token}` をルーティングする。未ログインで招待を開いた場合は `/login?from=<invite>` に退避し、ログイン後に参加確認画面へ復帰する（`from` は `/invite/` 内部パスに限定しオープンリダイレクトを防ぐ）。初回実装でDeep Linkの実機確認が難しい場合は、トークン入力/貼り付けで参加する画面（設定＞つながり＞「招待リンクで参加」）を代替として用意する。

> **現時点の実装状況（Phase 1）**: プロフィール・フレンド・招待URLの**データ基盤**（`profiles`/`friendships`/`genba_invites` テーブル・RLS・RPC・pgTAP[静的]・ドメイン型・純粋ロジックのテスト）まで。共有メンバーの実体は既存 `genba_shares` を使う。各画面UI（プロフィール/フレンド/共有メンバー/招待参加）・Deep Link 配線・実機E2Eは後続Phase（decisions.md D-233・docs/follow-up-work.md）。

### 7.9 計画・推し活遠征旅程

現場詳細に「計画」タブを追加し、公演会場を中心に観光地、飲食店、ホテル、駅、空港等を日別・時系列で管理できるようにする。詳細仕様は[現場詳細「計画」タブ・推し活遠征旅程 仕様書](itinerary-plan-spec.md)を単一の真実とする。

基本要件:

- 手動スポット登録はGoogle API設定、通信状態、無料／プレミアム区分に関係なく常に利用可能にする
- スポットは施設名、カテゴリ、住所、座標、日付・時刻、複数URL、メモ、画像を持てる
- 公演会場・開場・開演・終演、既存の交通・宿泊、スポット、移動区間を同じ日別タイムラインへ表示する
- 既存の交通・宿泊は複製せず参照追加し、重複入力を避ける
- 日付未定、時刻未定、日跨ぎ、旅程タイムゾーンを失わず扱う
- 移動時間とユーザー指定の余裕時間から、次の予定や開場へ間に合わない可能性を示す
- 経路、運賃、公共交通情報を取得できない場合は手動入力できる
- 保存済み旅程はオフライン閲覧・編集可能とし、通信復旧後に同期する
- 旅程、住所、座標、予約URLはデフォルト非公開とする
- 「行きたい場所」を何も追加していない（=計画未作成の）状態でも、登録済みの公演会場・開場・開演・終演を計画タブの最初から表示する。公演枠には時刻に加えアーティスト名・公演タイトルも表示する
- 計画未作成の状態を閲覧するだけではDBに計画レコードを作成しない。追加ボタンから予定・スポットを追加する操作をした時だけ計画を作成する
- 現場詳細の概要タブには、開催当日かつ計画にユーザー追加項目（スポット・移動区間等）がある場合のみ、概要カードの下・Todo一覧の前に「次の予定」カードとその時刻を表示する。計画が未作成、または開催当日でない場合は何も表示しない
- スポットの場所入力は、Google検索と手入力を1つの入力欄に統合した一体型UIにする。文字を入力するとGoogleの候補が出て、候補を選んでも選ばず手入力しても登録できる。入力欄をGoogle用と手入力用に分けて画面を複雑にしない
- Google APIが無効・未設定・オフライン・上限・障害でも、同じ入力欄がそのまま手入力欄として動き、手入力だけでスポットを登録できる。候補選択時のみ施設名・住所・Place IDを保存する（Google由来値のうち永続保存するのはPlace IDを主とし、名称・住所はユーザーが確認・保存した値として扱う）
- 住所入力欄には共有可否等の補足説明文（「共有時は規定で非公開として扱われます」等）を表示しない。旅程・住所・座標のデフォルト非公開方針は共有UI側で扱い、住所欄はラベルのみのシンプルな入力欄にする
- 移動経路の出発地・到着地の表示は、内部IDやenum（`spot`/`transport`/`lodging`/`note`）ではなく、ユーザーが読んで分かる表示名（施設名・交通名・宿泊名・メモ名、参照切れは「削除済みの…」）を使う
- 移動区間の通常UIには通貨・経路概要を出さない（DB項目としては保持）。金額欄は残し、日本円前提で「1,200円」のように表示する（通貨欄は表示しない）
- 移動区間の出発時刻の初期値は移動元の予定の終了時刻（無ければ開始時刻）、到着時刻の初期値は移動先の予定の開始時刻（無ければ終了時刻）とする。所要時間は手入力せず、出発・到着時刻から自動計算した読み取り表示（例「所要 約35分」）にし、保存時に自動計算値を保存する
- 移動区間カードの経路導線は役割を分ける。**「経路を確認」はアプリ内でGoogle Routesから経路を取得して表示する主要導線**（計画時刻＝区間の出発予定時刻ベース）で、公共交通では徒歩合計・路線・行き先・乗降駅・発着時刻・乗換・合計所要を乗換タイムラインとして表示する。**「最新の経路」は押した時点の現在時刻を出発時刻として、登録済みの出発地から再取得する**（当日・移動直前用。取得した経路で保存済み経路を更新するかは確認のうえ、承諾しなければ一時表示に留める）。**「Google Mapsで開く」は外部地図を開く補助導線**とし、出発地・到着地にPlace IDまたは緯度経度がある場合に加え、施設名・住所があるときも生成できるようフォールバックする（URLは Google 公式の経路形式 `dir/?api=1`・Place IDは`*_place_id`併記・移動手段付与）。対応アプリが無い端末でもブラウザで開き、開けないときは理由が分かる案内を出す
- スポットカード／スポット詳細に「地図で開く」導線を出す。Place IDがあればPlace ID、無ければ施設名・住所・緯度経度からGoogle Maps検索URLを生成して開く（手入力スポットでも施設名で開ける）。URL生成不能か起動失敗かを区別して案内する
- 会場・公演情報（開場/開演/終演）は計画の予定ではなく「目安」として、予定カードより小さく・薄く・「目安」が分かる見た目で表示する。予定として使う場合は「会場を追加」から予定スポットとして追加する
- 計画タブに「会場を追加」を置き、現場の会場をカテゴリ「ライブ・イベント会場」の予定スポットとして追加できる（訪問日=開催日・開始=開場・終了=終演）。会場未登録なら案内し、追加済みなら重複確認する

Google連携:

- 現場の会場入力も、スポットと同様にGoogle検索と手入力を1つの入力欄に統合した一体型UIにする。候補選択時に会場名・住所・Place IDを保存する（座標はField Mask対象外のため保存しない）。Google無効・未設定・通信不可でも手入力で保存できる。会場のPlace ID・住所はスポット同様に地図・経路導線へ使える
- Places／Maps／Routesを旅程MVPの主要機能とし、利用不能時は手動旅程へ縮退する
- Placesの基本取得はPlace ID・名称・住所・必要な帰属情報に限定し、電話、Webサイト、営業時間、写真、評価、レビュー、primary type、座標はMVPで取得しない
- Google（Places／Routes／Geocoding）へのリクエストは日本語優先・国内前提（`languageCode: ja`／`regionCode: JP`）で行い、施設名・住所・経路案内は日本語表記を優先して取得・表示する。距離はメートル法（`units: METRIC`）で扱う
- ja/JP対応より前に保存された既存データ（ユーザーが手入力・またはGoogle候補選択で保存した施設名・住所）は、アプリが自動で再取得・一括上書きしない。ユーザーが保存した値を尊重する。既存スポット・会場を開いても住所をGoogleで自動再取得しない
- 既存データを日本語表記へ更新したい場合は、ユーザーの明示操作でのみ行う。スポット・会場編集画面でGoogle候補を選び直す（「明示変換」）ことで施設名・住所を差し替えられる。将来「Googleの情報で更新」導線を設ける場合も、更新前に確認ダイアログを出し、ユーザーの同意なしに置き換えない
- リポジトリ同梱のデモデータ・seed・fixture・サンプル・オンボーディング固定データの住所は日本語表記に揃える（英語表記の日本語住所を残さない）。実ユーザーの端末ローカルデータはこの限りではなく自動変更しない
- APIは最小Field Mask、Autocomplete session token、debounce、single-flight、明示操作、ユーザー別クォータ、予算通知、kill switchで料金を制御する
- Web Service用キーをアプリへ埋め込まず、認証済みサーバー境界から呼び出す
- Place IDは永続保存・再利用できるが、Google由来の施設名・住所・経路結果をAPI代替目的のユーザー横断恒久キャッシュへ保存しない
- 共有施設・共有概算経路はユーザー入力、施設提供、オープンデータ、契約データ等、保存・再利用権限を説明できる情報源で構築する
- 通常表示は保存済みの権利確認済み概算経路を優先し、Google Routesは「経路を確認」または「最新の経路」を押した場合だけ呼ぶ
- **現時点では経路検索（「経路を確認」「最新の経路」）は全ユーザーが利用できる**。アプリ内にプレミアム／ノーマルの正式なアカウント区分・課金導線が無いため、現仕様ではプレミアム限定にしない。UIに「プレミアム限定」文言を出さない
- 将来的に経路検索をプレミアム化する余地は残す。プレミアム制限は Edge Function の `ROUTES_REQUIRE_PREMIUM=true`（既定 `false`）の環境でのみ有効化し、そのときだけ `has_premium_routes_entitlement` を検証する。既定（未設定/false）では認証済みユーザーなら利用できる
- Google API 使用量は、プレミアム制限ではなくレート制限・kill switch・timeout・最小Field Mask・明示操作で制御する（プレミアム区分は費用制御の手段にしない）
- 計画段階で取得した経路は、ユーザーが確認のうえ保存できる。ただしGoogleの利用規約を尊重し、乗換ステップ・路線・生レスポンス等のGoogleコンテンツは恒久保存せず、ユーザーが自分の概算として確定した最小限の表示用データ（所要時間・距離・地図URL・確認日時）だけを`user_provided`として`ItineraryLeg`へ保存する（明示変換。Placesの名称・住所と同じ扱い）。APIキー・秘密情報は保存しない
- 「最新の経路」は現在時刻を出発時刻として登録済み出発地から取得し、保存済み経路を更新するか確認する。承諾しなければ画面上の一時表示に留める
- 全ユーザーが保存済み概算経路を閲覧でき、運賃は取得不能時や実態と異なる場合に手動入力・修正できる
- Google由来情報には必要なGoogle Maps／第三者帰属を表示し、ユーザーの明示操作なしに定期再取得しない

旅程MVPは手動登録、日別タイムライン、既存交通・宿泊の参照、手動移動区間、余裕時間、ユーザー画像、複数URL、オフライン同期に加え、最小Google Places、アプリ内地図、権利確認済み概算経路、Google Routesの明示的な最新取得を含む。実装は依存順に分割するが、Phase 4のRoutes完了までは旅程MVP完成と判定しない。Google連携の未設定・障害・上限到達時は手動機能へ縮退する。Google Place写真、共同編集、通知、AI旅程提案、混雑予測、複数スポット最適化はMVP対象外とする。

## 8. 思い出

### 8.1 移行

公演終了後の現場は過去の「思い出」として表示する。未来と過去を明確に分けるため、翌日以降は現場一覧と通常ホームには表示しない。

移行時も現場IDと登録済み情報は維持する。ユーザー操作による復元は不要だが、日付誤入力を修正した場合は表示区分を再判定する。

### 8.2 段階入力

標準導線は次の3段階とする。

#### 終演直後

- 写真を追加
- 短い感想を入力
- 特によかった曲を入力

#### 終演後

- 感想本文
- MC・当日メモ
- セトリ
- 座席・見え方

#### 後日

- 写真整理、表紙、タグ
- グッズ・戦利品
- 周辺で行った場所、食べたもの
- 記録の追記

ユーザーは終演前を含め、いつでも終演後・後日向け項目を先行入力できる。段階は入力制限ではなく、標準表示順と入力案内を決定するために使用する。これは入力画面の段階名であり、思い出へ移行する日時判定ではない。

短い感想は独立した必須データにせず、感想本文の初期入力として扱い、後から同じ欄を加筆できる。

### 8.3 入力通知

未入力通知は、該当項目が空の場合のみ送る。すべての項目の入力は必須とせず、ユーザーが「今回は入力しない」を選択した項目には再通知しない。

標準通知:

- 終演後: 写真または短い感想の入力案内
- 翌日: 「終演後」段階の項目がすべて未入力の場合に案内
- 3〜7日後: 後日項目がすべて未入力の場合に1回だけ案内

通知から対象入力画面へ直接遷移する。通知頻度と項目は設定で変更可能とし、同一現場について同種通知を繰り返さない。

### 8.4 思い出詳細

- 概要
- 写真
- 感想
- セトリ
- MC・当日メモ
- 座席・チケット
- グッズ・戦利品
- 共有メンバー
- お気に入り
- 表紙写真

感想はユーザーごとに保持し、共同編集しない。

#### 思い出アルバム（写真・関連項目）

- 写真の保存元は `MemoryPhoto` に一本化し、画面ごとに複製しない。グッズ・行った場所・
  食べたものの写真も同一テーブルに保持し、`subject_type`/`subject_id` で関連項目へ、
  `album_category`（当日の写真 / グッズ・戦利品 / 行った場所 / 食べたもの）で分類する。
- アルバムは分類で絞り込め、表紙（`is_cover` 優先、無ければ先頭）を持つ。
- 関連項目（グッズ・行った場所）を削除しても、既定では写真をアルバムへ残す。
  そのため `subject_id` は緩い参照とし外部キー制約を張らない。
- 写真を関連項目へ紐づける際は、同一 owner+genba の項目が実在することを Repository と
  DB の双方で検証し、孤立した `subject_id` を作らない。

データ整合（データ設計と同期）:

- Domain `MemoryPhoto` に `albumCategory`/`subjectType`/`subjectId` を追加。DB は Drift
  v12→**v13**（`memory_photos` に3列を追加し、既存写真を `album_category='event'` へ移行、
  過去 migration は変更しない追加方式）、Supabase **0019**（同3列＋分類/種別 CHECK＋索引、
  RLS は owner ベースの既存ポリシーを継承、apply_mutation の列許可は information_schema から
  動的取得のため新列は自動対象）。Mapper はコード⇄enum を相互変換する。

## 9. マイ推し

グループ／アーティスト単位で登録し、必要に応じてメンバーを自由入力する。

- グループ名、種別、カラー、メモ
- グループ画像、お気に入り
- メンバー名、推し区分、メンバーカラー、推し始めた日、誕生日、メモ、画像
- 推し区分: 最推し、推し、ゆる推し、箱推し、気になる
- 現場と思い出への紐づけを維持する

初回現場登録時に新しい推しを同時作成できるようにし、事前のマイ推し登録を必須にしない。

## 10. ユーザー投稿型公演DB

### 10.1 基本方針

全ユーザー共通の公演マスターをユーザー投稿で拡充する。ユーザーは既存公演を選択するか、新規追加できる。

表示は「参戦予定」ではなく「登録者」を使用する。

```text
登録者 124人
```

この数値は、その公演を自分の現場として登録したユーザー数を示し、実際の参加や出席を保証するものではない。

### 10.2 公演登録

必須項目:

- グループ／アーティスト
- 公演名
- 会場
- 日付
- 開演時間

類似判定にはグループ、日付、会場、開演時間を使用し、重複候補を登録前に表示する。

### 10.3 公開とプライバシー

- 公演マスター情報は共通情報として公開する
- チケット、座席、交通、宿泊、Todo、メモは公開しない
- 登録者数は匿名の集計値としてのみ表示する
- 自分の登録を人数集計に含めるかはプライバシー設定で選択可能にする
- 少人数時の特定リスクを避けるため、必要に応じて閾値未満は具体人数を表示しない

### 10.4 信頼性と運用

- 完全一致、登録者数、日付の近さを候補順に使用する
- 作成者による編集後、複数ユーザーが利用中の公演に影響する場合は変更内容を記録する
- 重複統合、誤情報の通報、管理者修正に対応する

## 11. 通知

通知対象:

- 現場が近づいたとき
- 期限の近い未完了Todoがあるとき
- 必要と回答した交通・宿泊が未登録のとき
- 終演後、翌日、後日の思い出入力
- 現場、初現場、推し始めた日の記念日
- 共有内容の更新

通知UX:

- 通知許可を求める前に用途を説明する
- 初期設定は重要通知に絞る
- 通知から対象画面へ直接遷移する
- 完了済み、不要指定済みの項目は通知しない
- 同日中の通知をまとめ、連続通知を避ける
- 種類別に有効／無効とタイミングを設定可能にする

## 12. マネタイズ

- Todoや持ち物に関連する場合のみ商品導線を表示する
- アフィリエイトであることを明示する
- 当日ホームの主要操作より広告導線を優先しない

## 15. 非機能要件

### 15.1 対応環境

- iOS／Androidのスマートフォンを対象とする
- レスポンシブな文字サイズと主要なアクセシビリティ設定に対応する
- OSのサポート対象バージョンは実装開始時に確定する

### 15.2 セキュリティ・プライバシー

- ユーザー単位のデータ分離
- 通信と保存データの暗号化
- 画像への認可付きアクセス
- 共有権限のサーバー側検証
- チケット画像や予約番号をログへ出力しない
- アカウント削除と関連データ削除
- デフォルト非公開

### 15.3 パフォーマンス・可用性

- ホームはキャッシュを先に表示し、バックグラウンド同期する
- 当日の重要情報はオフライン閲覧可能にする
- 写真アップロードは非同期・再試行可能にする
- 一覧はページングする
- 公演候補はインクリメンタル検索する
- 入力は自動保存し、通信復旧後に同期する

### 15.4 アクセシビリティ

- 色だけで状態を区別しない
- 推しカラー使用時も文字コントラストを確保する
- 主要操作に読み上げ用ラベルを設定する
- タップ領域を十分に確保する

## 16. 開発・リリース前提

### 16.1 クライアント

Windowsを主開発環境としてiOS／Androidを同一コードベースで開発するため、Flutterを推奨構成とする。

- 言語: Dart
- UI: Flutter
- 状態管理、ルーティング、ローカルDBは技術設計工程で確定する
- Push通知: Firebase Cloud Messaging、iOSはAPNs連携
- 写真選択、ディープリンク、オフラインキャッシュに対応する

### 16.2 Windowsでの開発範囲

Windowsで実施可能:

- 共通コードとAndroid版の開発
- Androidビルド、エミュレーター、実機テスト
- 単体テスト、ウィジェットテスト
- バックエンド開発

macOSが必要:

- Xcodeを用いたiOSビルド
- iOSシミュレーター・実機での最終確認
- 証明書、署名、プロビジョニング
- App Store Connectへの提出

iOSリリース工程では、Mac実機またはmacOS CIを利用する。

## 17. 成功指標

主要指標は「2件目の現場登録率」とする。

補助指標:

- 初回起動から最初の現場登録完了までの時間
- チュートリアル開始・完了・スキップ率
- 登録後24時間以内にTodoを1件以上利用した割合
- 公演当日にアプリを開いた割合
- 当日ホームの主要操作利用率
- 終演後7日以内に写真または感想を残した割合
- 通知から入力を開始・完了した割合
- 30日および次回現場時の再利用率

## 18. 最重要思想

本アプリの中心は旅行ではなく「現場」である。

```text
公演前: 準備するもの
当日: 確認するもの
終了後: 思い出として残るもの
数年後: 推し活人生の一部として見返すもの
```

本アプリの価値は「ひとつの現場を、予定から思い出まで育てること」に集約する。

## 19. UI・ビジュアルデザイン

UIは[UI・ビジュアルデザイン仕様書](design-spec.md)に従い、コンセプトを「夜明け前の遠征ノート」とする。

必須方針:

- やわらかな紫、白、十分な余白、カード型レイアウトを基調とする
- ホームは次の現場と残日数をヒーローカードで最優先表示する
- 現場詳細はヒーロー領域と、概要／Todo・持ち物／チケット／交通／宿泊／計画／メモのタブで構成する
- 思い出一覧と詳細は写真を主要な視覚要素とし、写真なしでも成立する
- 思い出一覧は、すべて／参戦済み／お気に入りで切り替え可能にする
- 思い出詳細は、感想／セトリ／グッズ／メモで内容を整理する
- マイ推しはプロフィール、メンバー、活動統計、次の現場、記念日を一画面で把握可能にする
- 設定は階層型リストとし、ライト／ダーク、推しカラー、プレビューを提供する
- Primaryカラーと推しカラーを区別し、推しカラーは小面積の個人化表現に限定する
- 写真上の文字コントラスト、文字拡大、読み上げ、48dp以上のタップ領域を保証する
- ユーザーが利用権限を持たないアーティスト写真や公演写真をアプリへ同梱しない
- お気に入り、参戦済み、表紙、推し画像、現場数・思い出数・参戦数を固定ダミーではなく保存データから表示する
- デザイン画像とのピクセル一致ではなく、情報階層、操作導線、コンポーネント、配色の一貫性を受入基準とする
