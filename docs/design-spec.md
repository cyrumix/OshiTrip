# UI・ビジュアルデザイン仕様書

版: 1.0  
基準: ユーザー提示の6画面コンセプト画像（2026-07-02）  
関連: [要件定義書](requirements.md) §19 / [アーキテクチャ](architecture.md)

## 1. デザインコンセプト

コンセプト名は**「夜明け前の遠征ノート」**とする。

ライブ前夜のわくわく、遠征の高揚感、終演後の余韻を、やわらかな紫、淡い光、余白、写真で表現する。かわいさだけに寄せず、移動中や当日の焦りの中でも情報を素早く読める、静かで上品な実用UIを目指す。

デザインの中心原則:

- 1画面1目的を明確にし、重要情報を上から順に配置する
- 情報は白いカードへ整理し、背景との階層を影と余白で表現する
- 紫は主要操作、選択状態、残日数、現在地の強調に限定する
- 公演写真は思い出の主役として大きく扱い、管理情報より視覚的に優先する
- 推しカラーは個人化用アクセントとし、本文や重要操作の可読性を壊さない
- 状態は色だけでなく、アイコン、ラベル、形、文言を併用して示す

## 2. カラートークン

基準色は次のとおりとする。実装では`ColorScheme`と意味ベースのDesign Tokenへ変換し、画面内へ色コードを直書きしない。

| トークン | 基準値 | 用途 |
|---|---:|---|
| Primary | `#7B5CFF` | 主要ボタン、選択タブ、FAB、残日数、フォーカス |
| Primary Light | `#EEE9FF` | ヒーローカード、選択背景、淡いグラデーション |
| Background | `#F8F6FC` | ライトテーマの画面背景 |
| Surface | `#FFFFFF` | カード、シート、入力領域 |
| Text Primary | `#29212E` | 見出し、本文の主テキスト |
| Text Secondary | `#7D7788` | 補足、日付、メタ情報 |
| Divider | `#E9E5F0` | 区切り、未選択タブ、カード境界 |
| Error | Materialのerror基準 | 期限超過、危険操作。紫で代用しない |
| Oshi Accent | ユーザー選択 | 推し単位の左罫線、アバターリング、小面積の装飾 |

ライトテーマでは白と淡紫を中心にし、ダークテーマは同じ情報階層を暗色面へ変換する。ダークテーマをコンセプト画像の色反転だけで作らず、WCAG AA相当の文字コントラストを維持する。

## 3. タイポグラフィと余白

- 日本語本文の基準は14〜16sp。補助情報は12sp未満にしない
- 画面タイトルは18〜20sp、カード見出しは16〜18spを基準とする
- 残日数などの重要数値は28〜40spで、単位との強弱を付ける
- 文字ウェイトはRegular / Medium / Boldの3段階を基本とする
- 4dpグリッドを使用し、標準余白は8 / 12 / 16 / 24dpとする
- カード角丸は12〜16dp、主要ヒーローカードは16〜20dpを基準とする
- 主要タップ領域は48×48dp以上とする
- OS文字拡大時にテキストを切らず、カード高さを固定しすぎない

## 4. 共通コンポーネント

次を共通Design Systemとして実装し、画面ごとの類似Widget乱立を避ける。

- `AppScaffold`: 背景、SafeArea、AppBar、Bottom Navigationの共通骨格
- `AppCard`: 標準カード、押下可能カード、警告カード
- `HeroEventCard`: 写真または淡紫グラデーションを背景にした次の現場カード
- `EventListCard`: 左側の推しカラー罫線、日付、会場、公演名、残日数を持つカード
- `StatusIconItem`: アイコン、短いラベル、状態値を縦に表示
- `SectionHeader`: 見出し、件数、右側アクション
- `SegmentTabs`: 選択中を紫の文字と下線で示すタブ
- `PhotoMemoryCard`: サムネイル、日付、公演名、会場、記録件数、お気に入り
- `OshiAvatar`: 写真、イニシャルフォールバック、推しカラーリング、選択状態
- `CountStat`: 現場数、思い出数、参戦数等の数値表示
- `SettingsRow`: アイコン、項目名、現在値、遷移矢印
- `EmptyState`: 短い説明と次の1アクション
- `LoadingSkeleton`: カード配置を保ったスケルトン
- `SyncBadge`: 端末保存済み、同期中、失敗、競合を必要時だけ表示
- `FavoriteButton`: お気に入り状態をハートの形とラベル／Semanticsで表現

影は控えめにし、カードの境界は背景差、1px相当の境界、弱い影を組み合わせる。すべてのカードに強いドロップシャドウを付けない。

## 5. グローバルナビゲーション

画面下部に次の5タブを固定する。

```text
ホーム / 現場 / 思い出 / マイ推し / 設定
```

- アイコンと日本語ラベルを常時表示する
- 選択中はPrimary、未選択はText Secondaryで示す
- タブ切替後も各タブのスクロール位置と内部ナビゲーションを保持する
- 新規登録はホームまたは対象一覧の右下FABで提供する
- FABが一覧の最終項目やBottom Navigationを覆わない余白を確保する

## 6. ホーム画面

### 6.1 ヘッダー

- 中央に`HOME`または日本語の「ホーム」を表示する
- 右側に通知入口を置く。通知未実装期間は押せないベルを表示しない
- 冒頭に「次の現場まで」を表示する

### 6.2 次の現場ヒーローカード

- 最も近い未来の現場を大きな紫系カードで表示する
- 残日数を最も大きく表示する
- 日付、開場／開演、会場、公演名を同一カード内に表示する
- 公演写真がある場合は、文字可読性を保つ暗めのオーバーレイを付けて背景へ使用できる
- 写真がない場合はPrimary〜Primary Lightのグラデーションと抽象的な光表現を使用する
- カード下部にTodo、交通、宿泊、チケットの状態ショートカットを4分割で置く
- 各状態は「3件」「未登録」「登録済み」等の短い値を表示する

### 6.3 今後の現場

- 日付順の縦カード一覧とする
- 各カードへ日付、会場、公演名、残日数、推しカラー罫線を表示する
- 最も近い現場と同一情報を過剰に重複表示しない
- 右下に現場追加FABを置く

当日はヒーローカードを当日ホームへ切り替え、情報量が増えても最上位操作を隠さない。

## 7. 現場詳細画面

### 7.1 ヒーロー領域

- 公演写真または紫系フォールバック背景を横長に表示する
- 公演名、日付、開場／開演、会場、残日数を重ねる
- 文字にはグラデーションオーバーレイを使用する
- 共有中は共有人数または共有状態を表示する

### 7.2 詳細タブ

```text
概要 / Todo・持ち物 / チケット / 交通 / 宿泊 / 計画 / メモ
```

- タブは横スクロール可能とし、選択状態を下線と文字で示す
- 概要には公演情報、同行者／共有メンバー、準備サマリ、次のアクションを表示する
- 開催当日かつ計画にユーザー追加項目がある場合のみ、概要カードの下・Todo一覧の前に「次の予定」カードを表示する。それ以外は表示しない
- Todo・持ち物は両セクションを分け、Todoは未完了数と総数、持ち物は未登録／未対応／準備OKを表示する
- 計画は公演を固定アンカーにした日別タイムラインを主表示とする
- 詳細項目の編集はタブ内またはBottom Sheetで行い、長大な1画面にしない
- 共有メンバーは小型アバターと「共有メンバー」ボタンで示す

### 7.3 計画タブ

- 上部に日付選択と「タイムライン／地図／候補」の表示切替を置く。MVPで未実装の地図は見せかけの操作として表示しない
- 公演会場・開場・開演・終演を動かせない強調アンカーとして表示する。アンカーには時刻に加えアーティスト名・公演タイトルも表示する
- 計画（行きたい場所）を何も追加していなくても、上記アンカーは最初から表示する。この閲覧だけではDBに計画を作成せず、追加メニューからの追加操作時にだけ作成する
- スポット、交通、宿泊、移動区間を縦の時系列で結び、時刻、カテゴリ、所要時間を一読できるようにする
- 日付未定は候補、時刻未定は各日の末尾へ置き、情報を勝手に補完しない
- 移動時間と余裕時間が不足する場合は警告アイコンと「間に合わない可能性」を併記する
- 追加メニューでは「自分で入力」を常に表示し、Google連携の利用可否に依存させない
- スポットカードはカテゴリ画像またはユーザー画像を使用する。Google Place写真はMVPで取得しない
- 交通・宿泊の取り込みでは登録済み／追加済み／参照切れを文字で区別する
- オフライン時もタイムライン、ユーザー入力住所、権利確認済みの保存済み概算経路を隠さない。Google検索と最新経路取得だけを停止する
- 320pt幅では1列、横向き・大画面ではタイムラインと地図の2ペインを許容する
- Dynamic Type 200%では時刻列を固定幅にせず、カード内で安全に折り返す

## 8. 思い出一覧画面

- 上部に「すべて／参戦済み／お気に入り」のSegment Tabsを置く
- 右上に思い出追加または写真追加の入口を置く。ただし現場と無関係な孤立データを作らない
- 1カードに写真サムネイル、日付、公演名、会場を表示する
- 写真枚数、セトリ有無、感想有無を小型アイコンと数値で示す
- 右上にお気に入りボタンを置く
- 写真がない場合は、推しカラーと公演情報を用いたプレースホルダーを表示する
- 一覧は写真を主役にしつつ、文字拡大時にも公演名と日付を失わない

「参戦済み」は実際の出席をユーザーが明示した場合だけ使用する。公演DBの「登録者」と混同しない。

## 9. 思い出詳細画面

- 最上部に写真カルーセルを大きく表示し、現在位置を`1/18`形式で示す
- 写真の下に日付、公演名、会場、写真枚数、セトリ、感想、共有状態を表示する
- お気に入り操作を提供する
- 内容は「感想／セトリ／グッズ／メモ」のタブで整理する
- 感想は日記カードとして読みやすい行間と余白で表示する
- 写真がない場合も、感想やセトリを主役にできるレイアウトへ縮退する
- 編集入口と閲覧状態を明確に分ける

## 10. マイ推し画面

- 推しグループ／アーティストごとのプロフィールカードを上部へ表示する
- グループ名、種別、最推しメンバー、推しカラー、お気に入り状態を表示する
- メンバーは横スクロールのアバター列とし、選択中または最推しをリングで示す
- 現場数、思い出数、参戦数等の統計を3分割カードで表示する
- 「次の現場」を淡紫のカードで表示し、残日数を強調する
- 誕生日・記念日を近い順に表示する
- 写真未登録時はイニシャルまたは汎用アイコンを使用する

統計の名称は実データの意味と一致させる。「参戦数」は参加をユーザーが明示した場合のみ使用し、単なる登録数を参戦数と表示しない。

## 11. 設定・テーマ設定

設定トップは次を階層型リストとして表示する。

```text
アカウント設定
通知設定
テーマ設定
推しカラー設定
プライバシー・共有
データ管理
ヘルプ・お問い合わせ
ログアウト
```

- 危険操作は通常項目と視覚的に分離する
- テーマ設定ではライト／ダークのプレビューカードを並べる
- 推しカラーは複数のプリセット色とカスタム色を扱う
- 選択色を現場カードの罫線、アバターリング、プレビューへ即時反映する
- 色選択だけに依存せず、選択リングとチェックを表示する
- 変更は自動保存し、キャンセルが必要な設定では明示的な保存操作を置く

## 12. 写真・画像の扱い

- 公演写真、思い出写真、推し画像はユーザーが利用権限を持つ画像だけを登録する
- 特定アーティストや公演の画像をアプリへ無断同梱しない
- サンプル／テストでは権利処理済み素材、抽象グラデーション、プレースホルダーを使用する
- サムネイルと原画像を区別し、一覧で原画像を毎回展開しない
- 読み込み中、失敗、オフライン、権限喪失、削除済みの各状態を用意する
- 写真の上へ文字を載せる場合は、オーバーレイ後のコントラストを自動または固定規則で保証する
- センシティブなチケット画像を公演ヒーローや思い出一覧のサムネイルへ自動使用しない

### 12.1 デザインに必要なデータ

画像内の表示を固定値やダミーで作らず、次をドメインモデルと永続化へ反映する。

- 現場ヒーロー画像: ユーザーが明示的に選んだ公演用画像。チケット画像とは別フィールド・別用途とする
- 思い出表紙: `MemoryPhoto.isCover`を使用し、未指定時は安全な規則で最初の思い出写真を候補にする
- お気に入り: 思い出単位の`isFavorite`を保持し、一覧・詳細の両方から変更可能にする
- 参加状態: `planned / attended / notAttended / canceled`等の明示状態を持ち、「参戦済み」は`attended`だけを対象にする
- 推し画像: グループ画像とメンバー画像を、イニシャルフォールバック付きで保持する
- 活動統計: 現場数、思い出数、参戦数は保存済みデータから導出し、固定値として保持しない
- 次の現場: 対象の推しと紐づく未来の現場から最も近い1件を導出する
- 誕生日・記念日: メンバーの誕生日、推し始めた日、ユーザー定義記念日から導出する

画像本体のStorage同期が未完成のPhaseでは、ローカル永続参照と安全なプレースホルダーで機能させる。利用不能なアップロードを成功済みとして表示しない。

## 13. モーションとフィードバック

- タブ切替、カード展開、保存完了は150〜250ms程度の控えめなアニメーションとする
- Reduce Motion設定を尊重する
- FABやお気に入り操作は即時反映し、失敗時に状態を戻して理由を示す
- 自動保存は常時トーストを出さず、必要な場面だけ同期状態を表示する
- スケルトンから実データへの切替で大きなレイアウトシフトを起こさない

## 14. レスポンシブ・アクセシビリティ

- 基準幅は360〜430dpのスマートフォンとする
- 小型端末、横向き、文字200%で主要操作と情報が欠落しない
- タブが収まらない場合は横スクロールまたは適切な別ナビゲーションを使用する
- 画像へ代替説明を設定できる構造にする。装飾画像は読み上げ対象外にする
- アイコンボタンにはTooltipとSemantics labelを設定する
- お気に入り、完了、同期、準備状態を色だけで表現しない
- ライト／ダークの双方で主要テキストのコントラストを検証する

## 15. 受入条件

- ホーム、現場詳細、思い出一覧、思い出詳細、マイ推し、設定／テーマの6領域が本書の情報階層に従う
- ヒーロー画像、お気に入り、参加状態、表紙、推し画像、統計が実データへ接続され、固定ダミー表示ではない
- Primaryと推しカラーの責務が混在していない
- 写真なしでも全主要フローが成立する
- 写真ありでは画像が主要な視覚要素になり、文字が読める
- 主要画面のloading / empty / error / offline / data状態が揃う
- ライト／ダーク、推しカラー8色以上、文字拡大、狭幅でWidgetまたはGolden testがある
- Golden testはOSフォント差で不安定にならない範囲に限定し、意味上のWidget testと併用する
- コンセプト画像との完全なピクセル一致ではなく、情報階層、コンポーネント、色、余白、操作導線の一致を合否基準とする
