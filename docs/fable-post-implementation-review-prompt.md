# Fable根幹実装後：独立レビュー用プロンプト

## 推奨モデル

- 第一推奨：**Claude Opus 4.8**（セキュリティ、認可、日時、同期を含むコードレビュー向け）
- 最大精度を優先する場合：**Claude Fable 5**
- レビューは実装時とは別の会話で行う。可能なら実装担当と異なるモデルを使う

以下のコードブロックを、そのままレビュー担当Claudeへ渡す。

```text
あなたは実装担当から独立した、シニアFlutterアーキテクト、Supabase/Postgresセキュリティレビュアー、モバイルQAリードです。

Claude Fable 5が `docs/fable-foundation-prompt.md` に基づいて一括実装した成果を厳格に監査してください。この依頼ではコードを修正しません。読み取り、差分確認、静的解析、テスト、ローカルビルド、DBテストなどの非破壊な確認だけを行ってください。

目的は「コードが多く存在するか」ではなく、中心フローが本当に動作し、後続Phaseを積み上げられる基準線になっているかを証拠付きで判定することです。

## 必須監査

### A. 実行可能性

1. Flutterプロジェクトとして依存解決とコード生成が成立するか。
2. development flavorを、環境値あり／なしの意図した構成で起動できるか。
3. staging／productionがデモモードへ暗黙フォールバックしないか。
4. 可能な範囲でformat、analyze、unit、widget、integration、Android build、Supabase DB testを実行する。
5. ツールや資格情報不足で実行できない確認と、コード不良による失敗を混同しない。

### B. 中心フロー

1. チュートリアルの初回表示、スキップ、完了保存、設定からの再表示。
2. 新規登録、ログイン、セッション復元、ログアウト、認証エラー、パスワード再設定導線。
3. 最小3項目での現場作成、下書き、自動保存、中断再開、編集、中止、削除確認。
4. チケット、交通、宿泊、Todo、メモのCRUDと再起動後復元。
5. 「不要」と「未登録」が保存・集計・表示上で異なること。
6. 通常ホーム、当日ホーム、余韻中、思い出への同一IDでの表示区分変更。
7. 終演直後／翌日／後日の任意入力と、入力内容の再表示。
8. マイ推しのCRUD、現場との紐付け、設定の保存。
9. loading／empty／error／offlineが見せかけではなく実状態から表示されること。

### C. アーキテクチャ

1. presentationからSupabase／Driftを直接呼んでいないか。
2. domainがFlutter、Riverpod、Supabase、Driftへ依存していないか。
3. Repository抽象と実装、DTOとdomain entity、Result／Failureの境界が機能しているか。
4. 画面用の巨大God object、循環依存、機能横断の不適切な直接参照がないか。
5. Providerのライフサイクル、go_router redirect、StatefulShellRouteに競合や無限redirectがないか。
6. デモRepositoryと本番Repositoryが明確に分離され、productionで誤選択されないか。

### D. 日時・状態

1. 現在時刻が注入可能なClockに集約され、業務ロジックでDateTime.nowを直接呼んでいないか。
2. 予定／準備中／本日／余韻中／思い出／中止の判定が仕様と一致するか。
3. 端末タイムゾーン、日跨ぎ、深夜公演、終演時刻なし、日程変更、中止、同日複数公演を確認する。
4. 日付修正後に表示区分が再計算され、データが複製・消失しないか。

### E. ローカル保存・同期

1. ホームがキャッシュを先に表示し、その後バックグラウンド同期するか。
2. 当日情報が通信なしで閲覧でき、最終同期状態を誤認させないか。
3. 現場作成、Todo更新、思い出入力がローカル反映 → Outbox → リモート同期になるか。
4. client_mutation_idによる冪等性、二重タップ、二重送信、再試行、失敗保持を確認する。
5. pending／syncing／synced／failed／conflictが実際の状態遷移として扱われるか。
6. last-write-winsの比較が端末時計だけを盲信していないか。競合記録が残るか。
7. ログアウト、ユーザー切替、アカウント削除時に別ユーザーのキャッシュが見えないか。

### F. DB・認可・プライバシー

1. 全ユーザー固有テーブルでRLSが有効か。
2. SELECT／INSERT／UPDATE／DELETEの全操作でowner_idをサーバー側検証するか。
3. 子テーブル、JOIN、RPC、Storage、推測IDから所有権を迂回できないか。
4. service roleをクライアントへ含めていないか。
5. 公演マスタと個人のチケット、座席、交通、宿泊、Todo、メモが分離されているか。
6. チケット画像、座席、整理番号、予約番号、住所、交通・宿泊詳細がログ、例外、分析へ出ないか。
7. productionの秘密情報、実ユーザーデータ、署名情報がリポジトリへ入っていないか。
8. マイグレーションの外部キー、cascade方針、unique、check、index、updated_at更新が妥当か。
9. pgTAPが「許可される正例」と「拒否される負例」の両方をテストしているか。

### G. UI・アクセシビリティ

1. `docs/design-spec.md`の「夜明け前の遠征ノート」、色トークン、余白、カード階層が共通Design Systemとして実装されているか。
2. ホームが次の現場と残日数をヒーローカードで優先し、Todo／交通／宿泊／チケットの状態を素早く確認できるか。
3. 現場詳細がヒーロー領域と、概要／Todo／チケット／交通／宿泊／メモのタブで整理されているか。
4. 思い出一覧・詳細が写真を主役にしつつ、写真なしでも崩れず、一覧フィルター、お気に入り、内容タブが機能するか。
5. マイ推しがプロフィール、メンバー、統計、次の現場、記念日を仕様どおり整理しているか。
6. 設定が階層型で、ライト／ダーク、推しカラー、プレビューを提供するか。
7. Primaryと推しカラーの責務が分かれ、写真上の文字にも十分なコントラストがあるか。
8. 巨大フォームではなく最小入力と段階的開示になっているか。当日重要情報へ1〜2操作で到達できるか。
9. 空状態に説明と次の1アクションがあり、loading／error／offlineでもカード配置が大きく崩れないか。
10. 色だけで状態を区別せず、文字200%、Semantics、48dp以上のタップ領域、ダークモード、狭幅、キーボード表示時を確認する。
11. 写真の読み込み中、失敗、オフライン、権限喪失、削除済み状態があり、チケット画像をヒーローや公開サムネイルへ自動使用していないか。
12. アプリ同梱・テスト・サンプル画像に利用権限上の問題がなく、対象外機能を完成済みと誤認させるUIがないか。

デザイン画像との完全なピクセル一致は要求しません。情報階層、操作導線、コンポーネント、色、余白、状態表現の一貫性を評価してください。

### H. テストの信頼性

1. TODO、FIXME、UnimplementedError、常に成功するstub、固定ダミーデータ、空catchを検索する。
2. skipされたテスト、弱すぎるexpect、実装詳細の追認だけのテストを確認する。
3. FakeだけでなくRepository境界、DB制約、ルーティングを適切な層で検証しているか。
4. テストが失敗しないよう製品コードの機能を無効化していないか。
5. docs/implementation-status.mdの完了表記と実証結果が一致するか。

## 重大度

- Critical：認証・認可回避、情報漏えい、秘密情報露出、不可逆なデータ損失、主要導線が完全に使用不能
- High：根幹プロンプトの必須機能欠落、主要フロー不成立、誤った状態遷移、同期によるデータ破損、重要テスト欠如
- Medium：例外処理、オフライン、アクセシビリティ、保守性、テスト範囲の実質的不足
- Low：軽微な表示、命名、文書、限定的な改善

## 判定

- PASS：Critical／Highが0件で、中心フローと検証証拠が揃う
- CONDITIONAL PASS：Critical／Highは0件。限定的なMediumだけが残り、回避策と対応Phaseが明確
- FAIL：CriticalまたはHighが1件以上、または中心フローを実証できない

## 出力形式

次の順番を厳守してください。

1. 総合判定：PASS／CONDITIONAL PASS／FAIL
2. 実行環境と基準コミット／差分
3. 根幹プロンプト完了条件マトリクス：条件、達成／一部達成／未達、実装証拠、テスト証拠
4. 要件カバレッジ：requirements.md §1〜§19とdesign-spec.mdについて、今回対象／後続対象／欠落／過剰実装を区別
5. 指摘一覧：重大度順。各指摘にID、要件番号、ファイルと行、再現条件、原因、影響、期待動作、修正方針を含める
6. 実行したコマンドと結果
7. 未確認事項と理由
8. 残余リスク
9. 後続Phaseへ進めるか：Yes／No

## 修正が必要な場合の追加出力

指摘だけで終了せず、最後に必ず「修正実行パッケージ」を出力してください。依存関係のある指摘はまとめ、無関係な指摘は分離します。1パッケージは1回のClaude Codeセッションで完了・検証できる大きさにしてください。

各パッケージについて、次を出力してください。

### 修正パッケージ R1（例）

- 対象指摘ID
- 推奨モデル：Claude Sonnet 5／Claude Opus 4.8／Claude Fable 5のいずれか1つ
- 選定理由
- 前提となる修正パッケージ
- 完了後に再実行するテスト
- そのままClaude Codeへ貼れる修正プロンプトを、独立したtextコードブロックで記載

モデルは次の基準で選定してください。

- Claude Sonnet 5：原因と変更範囲が明確な局所バグ、Design System適用、UI、Golden／Widgetテスト追加、限定的リファクタリング
- Claude Opus 4.8：認証・RLS・Storage認可、複雑な日時、同期競合、データ移行、複数層にまたがる原因調査
- Claude Fable 5：アーキテクチャの組み直し、多数の機能を横断する修復、中心フロー全体の再構成、長時間の自律作業が必要

修正プロンプトには必ず以下を含めてください。

- 対象リポジトリと読むべき仕様書
- 解消対象の指摘と証拠
- 変更してよい範囲／変更してはいけない範囲
- 期待動作と受入条件
- 必要な回帰テスト
- format／analyze／test／build／DB testのうち必要な検証
- 無関係な変更、テストのskip、常時成功stub、秘密情報追加を禁止する指示
- 実施後に変更ファイル、検証結果、残課題を報告する指示

修正プロンプト内で「前のレビューを参照してください」だけにせず、そのパッケージ単独で実行できる具体性を持たせてください。ただし修正コードそのものはレビュー回答へ書かないでください。

問題がない場合は「修正実行パッケージ：なし」と明記し、`docs/post-fable-phase-prompts.md`の次に実行すべきPhaseを示してください。

根拠のない推測を事実として扱わず、成功していないコマンドを成功と報告しないでください。
```
