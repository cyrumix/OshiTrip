# Fable根幹実装後：Phase別実装プロンプト集

## 使い方

1. 最初に [Fable根幹実装後レビュー](fable-post-implementation-review-prompt.md)を実行する。
2. 現行コミット`7415653`については、[現行プログラム再レビューと修正実行プロンプト](current-program-review-and-remediation-prompts.md)のR1〜R8を依存順に実行する。別コミットではレビュー結果を更新してから使う。
3. 再レビューでCritical／Highが0件になってから、本書のPhase 1以降へ進む。
4. 各Phaseは新しいClaude Code会話で、共通プロンプトと対象Phaseプロンプトを続けて渡す。
5. 各Phase後にレビューを実行し、`requirements-traceability.md`へ実装箇所とテスト証拠を記録する。

既存実装を無条件に作り直さない。Fableが既に正しく実装した項目は検証して保持し、仕様との差分だけを実装する。

## モデルの使い分け

| 作業 | 第一推奨 | 理由 |
|---|---|---|
| 明確なUI、CRUD、テスト、局所改善 | Claude Sonnet 5 | 日常的なコーディングと反復に適する |
| RLS、共有、削除、同期、複雑な日時 | Claude Opus 4.8 | 深い推論とセキュリティ判断が必要 |
| 大規模な横断修復、最終全体収束 | Claude Fable 5 | 長い自律作業と多数の依存関係を扱う |

利用できない場合の代替は、Fable 5 → Opus 4.8 → Sonnet 5の順。ただし、セキュリティ境界をSonnetへ代替する場合は作業をDB／クライアント／テストに細分化し、独立レビューを必須とする。

## 全Phase共通プロンプト

各Phaseの先頭に、以下をそのまま渡す。

```text
あなたはこのリポジトリの実装担当です。Claude Fable 5による根幹実装を壊さず、指定されたPhaseの要件差分を実装し、検証してください。

最初に以下を全文読んでください。

- README.md
- docs/requirements.md
- docs/design-spec.md
- docs/fable-foundation-prompt.md
- docs/architecture.md
- docs/adr/ 配下すべて
- docs/requirements-traceability.md
- docs/implementation-status.md
- docs/decisions.md（存在する場合）
- docs/follow-up-work.md（存在する場合）
- docs/fable-post-implementation-review-prompt.md
- docs/current-program-review-and-remediation-prompts.md（現在の基準コミットに対応する場合）
- 現在のコード、テスト、DBマイグレーション、Git差分

実装原則:

1. requirements.mdを要件の正とする。承認済みADRと既存アーキテクチャを尊重する。
2. 最初に対象要件と既存実装を突合し、「既に達成／修正が必要／未実装」を短く整理してから変更する。
3. 既に正しく動く実装を作り直さず、今回のPhaseに必要な最小差分を実装する。
4. presentationからSupabase／Driftを直接呼ばない。domainをFlutterや外部SDKへ依存させない。
5. 認可の最終判断をクライアントUIに置かない。RLS、Storage policy、RPCで強制し、正負ケースを自動テストする。
6. 変更はローカル反映 → Outbox → リモート同期を維持し、通信失敗で入力を失わない。
7. チケット、座席、整理番号、予約番号、住所、交通・宿泊詳細、感想、画像URLをログや分析へ出さない。
8. loading、empty、error、offline、再試行、二重操作、競合、タイムゾーン、日跨ぎを対象にする。
9. 対象外機能を先行実装しない。TODOだけの画面や常に成功するstubを完成機能として追加しない。
10. ユーザーの既存変更を破棄せず、秘密情報をコミットしない。
11. 仕様上の軽微な曖昧さは安全側で決定してdocs/decisions.mdへ記録する。重大な製品判断だけを質問する。
12. 依存追加は必要最小限とし、Flutter stableとの互換性とライセンスを確認する。
13. UIはdocs/design-spec.mdのDesign Token、共通コンポーネント、画面別情報階層に従う。画面へ色コードや類似カードを個別実装しない。
14. 写真なしでも機能を成立させ、写真ありでは権利処理済み画像だけを使用する。チケット画像を公演ヒーローや一覧へ自動露出しない。
15. Primaryと推しカラーを分離し、ライト／ダーク、文字拡大、狭幅、Semantics、Reduce Motionを各Phaseで維持する。

各Phaseで必ず行うこと:

- 必要なmigrationと後方互換性を用意する。
- domain、repository/notifier、widget、DB認可のうち該当するテストを追加する。
- UI変更時は意味上のWidget testを必須とし、安定して検証できる主要コンポーネントに限定してGolden testを追加する。
- dart format、flutter analyze、flutter test、対象integration test、可能なbuild／supabase test dbを実行する。
- テストをskip・削除・弱体化して通さない。
- docs/requirements-traceability.mdとdocs/implementation-status.mdを、実証できた内容だけ更新する。
- 完了時に、変更ファイル、満たした要件、実行結果、未検証事項、残課題を簡潔に報告する。
```

## Phase 0：レビュー指摘の収束

推奨モデル：レビューが各修正パッケージに指定したモデル。指定がない場合、局所修正はClaude Sonnet 5、認可・同期・移行はClaude Opus 4.8、大規模再構成はClaude Fable 5。

レビュー結果に修正がある場合だけ実行する。レビューが生成した修正プロンプトを優先し、すべての修正後に再レビューする。判定がPASSまたはCONDITIONAL PASSになるまで、Phase 1へ進まない。

```text
Phase 0として、Fable根幹実装後レビューで提示された修正実行パッケージだけを、指定された依存順で実施してください。

- 各指摘の再現テストを可能なら先に追加する。
- 原因を直し、症状だけを隠さない。
- マイグレーション済みデータを破壊しない。
- 修正対象外の仕様変更や大規模整形を混ぜない。
- 各パッケージ完了時に対象テストを実行し、最後に中心フロー全体を回帰確認する。
- 全指摘について「解消／一部解消／未解消」を証拠付きで報告する。
```

## Phase 1：根幹機能の仕様完全化

対象：要件 §1〜§9、§13の基礎、§15.3・§15.4、§19のうち根幹機能に関係する差分。

推奨モデル：**Claude Fable 5**（根幹差分と6領域のDesign System適用を横断するため）。利用枠を抑える場合はClaude Sonnet 5で「Design Token＋ホーム」「現場詳細＋思い出」「マイ推し＋設定」の3作業へ分割する。

```text
Phase 1を実装してください。Fableが作った中心フローを再実装せず、requirements.md §1〜§9との未達差分をすべて閉じてください。

1. チュートリアルを最大4画面、スキップ可能、完了保存、設定から再表示可能にする。通知権限は要求しない。

4. チケットへ取得・当落・発券状態を含め、保存画像と外部URLを明確に分ける。
5. 交通／宿泊の未登録、必要、不要をデータ・UI・集計で区別する。
6. Todoテンプレートを現場種別、遠征有無、残日数に応じて提案し、適用前確認と不要項目の一括解除を可能にする。
7. 通常ホームカードへ仕様記載の全情報と項目別の準備重要度、次アクションを表示する。

9. 終演後の現場は同一IDのまま思い出へ移行し、日付訂正で再判定する。
10. 思い出の段階入力で、終演直後、翌日、後日の全項目を揃える。ユーザーはどの項目も先行入力でき、短い感想を本文へ加筆できるようにする。
11. セトリは曲順、曲メモ、アンコール、特によかった曲を扱う。写真は表紙・タグ、グッズと周辺の場所／食事も扱う。
12. マイ推しはグループとメンバーの全項目、各推し区分、画像、記念日、現場登録中の同時作成を扱う。事前登録を強制しない。
13. アカウント、テーマ、推しカラー、ダークモード、タイムゾーン、チュートリアル再表示を設定として保存・復元する。
14. 自動保存、再起動復元、オフライン表示、文字拡大、Semantics、コントラスト、タップ領域を全主要画面で検証する。
15. docs/design-spec.mdに従い、Design Tokenと共通コンポーネント（AppCard、HeroEventCard、EventListCard、StatusIconItem、SectionHeader、SegmentTabs、OshiAvatar、SettingsRow、LoadingSkeleton等）を整備する。
16. ホームを「次の現場」ヒーローカード、4状態ショートカット、今後の現場カード、追加FABの情報階層へ更新する。
17. 現場詳細を写真または紫系フォールバックのヒーロー領域と、概要／Todo／チケット／交通／宿泊／メモの横スクロールタブへ整理する。
18. 思い出一覧を写真サムネイル中心のカードと「すべて／参戦済み／お気に入り」タブへ更新し、写真枚数、セトリ、感想、お気に入りを表示する。
19. 思い出詳細を写真カルーセルと「感想／セトリ／グッズ／メモ」タブへ整理し、写真なしの縮退表示も実装する。
20. マイ推しをプロフィール、メンバーアバター、活動統計、次の現場、誕生日・記念日の構成へ更新する。実データにない統計を固定値で表示しない。
21. 設定トップを階層型リストへ更新し、ライト／ダークのテーマプレビューと推しカラー選択・プレビューの基礎を実装する。
22. 公演写真・推し画像がない場合は、淡紫グラデーション、推しカラー、イニシャルを用いたプレースホルダーを使用する。特定アーティスト画像を同梱しない。

- ホーム、現場詳細、思い出一覧・詳細、マイ推し、設定がdesign-spec.mdの情報階層を満たす。
- ライト／ダーク、推しカラー8色以上、360dp幅、文字200%、写真あり／なしのWidget testが通る。
- お気に入り、参加状態、表紙、ヒーロー画像、推し画像が再起動後も復元され、一覧フィルターと統計へ正しく反映される。
```

## Phase 2：ユーザー投稿型公演DB

対象：要件 §10、§15.3のページング・インクリメンタル検索。

推奨モデル：**Claude Opus 4.8**（公開データと個人データの境界、重複、匿名集計を含むため）。

```text
Phase 2として、requirements.md §10のユーザー投稿型公演DBを実装してください。

実装内容:

1. 現場登録時に既存公演をインクリメンタル検索して選択するか、新規公演を追加できるようにする。
2. 公演マスタの必須項目、任意項目、作成者、作成・更新日時、変更履歴を正規化する。
3. グループ／アーティスト、日付、会場、開演時間を使って登録前に重複候補を返す。表記正規化と完全一致を区別する。
4. 候補順は完全一致、登録者数、日付の近さを基本とし、理由をテスト可能な純粋ロジックにする。
5. 表示は「登録者○人」とし、参加や出席を保証しない説明を付ける。
6. 集計参加設定を尊重する。少人数閾値未満は具体人数を返さず、個人一覧を取得できるAPIを作らない。
7. 公演マスタは公開情報だけを持ち、チケット、座席、交通、宿泊、Todo、メモ、感想を別テーブル・別API境界に保つ。
8. 通報、管理者修正、重複統合、変更履歴の基盤を実装する。管理者判定はサーバー側で行う。
9. 一覧をページングし、検索のdebounce、空結果、低速通信、失敗、再試行を扱う。
10. 検索結果と重複候補はAppCard／EventListCard系コンポーネントで表示し、公演マスタの公開情報と個人の現場カードを視覚的にも混同させない。

受入条件:

- 既存選択と新規作成の両方から現場登録が完了する。
- 個人情報を公演マスタへ混入させる入力・RPC・直接DB操作を拒否する。
- 通常ユーザーは管理者修正・統合を実行できない。
- 重複候補、匿名集計、少人数マスキング、集計不参加をDB／unit／integration testで検証する。
- requirements.md §10.1〜§10.4へ証拠を記録する。
```

## Phase 3：現場・思い出共有と権限管理

対象：要件 §7.8、§8.4の共有部分、§13の共有・プライバシー設定、§15.2。

推奨モデル：**Claude Opus 4.8**。変更規模が非常に大きい場合はClaude Fable 5。

```text
Phase 3として、現場と思い出の共有を、サーバー側認可を中心に実装してください。

実装内容:

1. 招待、参加、拒否、解除、メンバー削除、権限変更、オーナー移譲を実装する。
2. 権限はowner／editor／viewer。viewerは閲覧のみ、editorは共有対象を編集可能、共有設定・削除・移譲はownerだけにする。
3. チケット画像、予約番号、住所、個人感想を含むセンシティブ項目を項目単位で共有可否設定でき、安全側を初期値にする。
4. 個人感想はauthor_id単位で保持し、他ユーザーとの共同編集を禁止する。
5. RLS、認可ビューまたは安全なRPC、Storage policyで権限を強制する。クライアント判定はUI補助だけにする。
6. Realtime更新を扱い、同時編集、削除競合、権限変更中、オーナー移譲中の整合性を保つ。
7. 共有解除・メンバー削除後に、ローカルキャッシュ、署名URL、画面履歴から情報へ到達できないようにする。
8. 招待、権限変更、共有解除、移譲を監査ログへ記録する。ログ本文にはセンシティブ値を含めない。
9. 設定へ共有範囲、検索可否、登録者数集計参加、項目別公開範囲を追加する。
10. 現場詳細の共有メンバーをOshiAvatarと同系統の小型アバターで表示し、共有中・招待中・権限喪失を色だけに依存せず示す。

受入条件:

- owner／editor／viewer／非メンバーごとにSELECT／INSERT／UPDATE／DELETE／Storage取得の許可表が自動テストされる。
- API直接呼出し、ID推測、子テーブル、期限切れ署名URLから権限を迂回できない。
- オーナー不在、複数owner、最後のowner離脱をDB制約またはtransactionで防ぐ。
- 感想は本人以外が更新できない。
- requirements.md §7.8、§8.4、§13、§15.2へ証拠を記録する。
```

## Phase 4：Push通知・入力リマインド・ディープリンク

対象：要件 §8.3、§11、§13通知設定、ADR-0009。

推奨モデル：**Claude Opus 4.8**（日時、冪等性、FCM/APNs、認証付き遷移を横断するため）。

```text
Phase 4として、FCM（iOSはAPNs連携）による通知基盤、サーバー側送信判定、設定、ディープリンクを実装してください。

実装内容:

1. 通知の価値を説明した後だけOS権限を要求する。拒否後も主要機能を利用可能にする。
2. FCMトークンの登録、更新、失効、ログアウト、複数端末、環境分離を扱う。トークンをログへ出さない。
3. 現場接近、Todo期限、必要だが未登録の交通／宿泊、終演直後、翌日、3〜7日後、記念日、共有更新を扱う。
4. 完了済み、不要指定、「今回は入力しない」は対象外にする。
5. 終演後は写真または短い感想、翌日は翌日対象がすべて空の場合、後日は後日対象がすべて空の場合に1回だけ案内する。
6. 同一現場・同一通知種別を冪等にし、同日通知を集約する。日程変更、中止、入力直後、タイムゾーン変更との競合を扱う。
7. 通知種類ごとの有効／無効とタイミングを設定画面で変更できるようにする。
8. 通知タップ時に認証状態を確認し、ログイン後も対象画面へ安全に復帰する。削除済み・権限喪失時は安全な案内へ遷移する。
9. 通知本文に座席、整理番号、予約番号、住所などを含めない。
10. ホーム右上の通知入口と設定の通知項目を実機能へ接続する。未実装・権限未決定時に見せかけのベルを表示しない。

受入条件:

- 送信判定を固定Clockでテストし、タイムゾーン、日跨ぎ、日程変更、中止、重複、入力競合を網羅する。
- Edge Function／cronの再実行でも重複送信しない。
- development／staging／productionのFirebase設定が混線しない。
- 通知拒否状態を含め、ディープリンクのrouting testが通る。
- 実送信できない環境ではstaging端末確認手順を残し、未確認を完了扱いにしない。
```

## Phase 5：画像・オフライン・同期の本番品質化

対象：要件 §6.2、§8、§15.2、§15.3の画像・キャッシュ・自動保存・再試行。

推奨モデル：**Claude Sonnet 5**。同期競合やマイグレーションの再設計が必要ならClaude Opus 4.8。

```text
Phase 5として、画像、オフライン、自動保存、同期を本番利用に耐える品質へ完成させてください。

実装内容:

1. 写真とチケット画像を認可付きSupabase Storageへ非同期アップロードし、進捗、キャンセル、失敗、再試行、二重登録防止を実装する。
2. アップロード失敗時にも文章や他項目を失わない。Outboxを再起動後も復元する。
3. 表紙、タグ、並び順、削除、ローカル参照、リモート参照、サムネイルを明確なモデルで扱う。
4. 署名URLの期限切れ、権限変更、共有解除、画像削除、アカウント削除を安全に処理する。
5. EXIFの位置情報等を保持するか除去するかを安全側で決定し、文書化する。
6. ホームのキャッシュ先行表示、当日重要情報のオフライン閲覧、最終同期時刻、古い可能性、キャッシュなしを明示する。
7. Outboxの指数バックオフ、接続回復、冪等性、順序依存、ユーザー切替、失敗隔離、競合記録を実装する。
8. ログアウトとアカウント削除後に保護キャッシュと画像を消去する。
9. 一覧ページング、画像メモリ使用量、低速通信、ストレージ不足を確認する。
10. 思い出一覧はサムネイルを使用し、詳細だけで原画像を展開する。ヒーロー画像のloading／error／offline／権限喪失用プレースホルダーを実装する。
11. 写真上の文字にはオーバーレイを適用し、ライト／ダーク双方でコントラストを検証する。センシティブなチケット画像はヒーローへ使用しない。

受入条件:

- オフライン編集 → アプリ再起動 → 接続回復 → 同期のintegration testが通る。
- 同一mutationや画像を複数回送っても重複しない。
- 未認可ユーザーはオブジェクトパスや古いURLを知っていても画像を取得できない。
- キャッシュなしと古いキャッシュを最新情報のように表示しない。
- センシティブ画像やURLがログ、分析、クラッシュ情報へ出ない。
```

## Phase 6：設定・データ主体の権利・マネタイズ・成功指標

対象：要件 §12、§13、§17、§15.2。非MVPのプレミアム機能は対象外。

推奨モデル：**Claude Opus 4.8**（データ出力・削除・共有所有権の整合性を含むため）。

```text
Phase 6として、設定を完成させ、データ出力・削除、MVPのマネタイズ導線、成功指標を実装してください。

実装内容:

1. 設定をアカウント、通知、表示、テーマ、推しカラー、ダークモード、タイムゾーン、プライバシー、人数集計、共有、チュートリアル、データ出力、削除として整理する。
2. 本人確認後に、ユーザー自身の現場、思い出、設定、画像一覧を機械可読形式で出力する。生成物は認可付き・期限付きにする。
3. アカウント削除で個人データ、画像、端末トークン、キャッシュを削除する。共有現場は所有権移譲／共有解除／削除の明示ルールに従いtransactionで処理する。

8. §17の主要指標「2件目の現場登録率」と補助指標を、匿名・最小限・重複なしで計測する。同意とプライバシー設定を尊重する。

10. 設定をdesign-spec.mdの階層へ完成させ、テーマプレビュー、推しカラーのプリセットと選択表示、データ管理、ヘルプ、ログアウト、危険操作の分離を実装する。
11. 推しカラー変更を現場カードの罫線、アバターリング、プレビューへ即時反映し、Primaryカラー自体は変更しない。

受入条件:

- 他ユーザーが出力ファイルを取得できず、期限後は本人も取得できない。
- 削除後にDB、Storage、Push、ローカルキャッシュから対象情報へ到達できないことをテストする。

- 分析イベントの重複、拒否設定、オフライン再送、禁止フィールドをテストする。
- §12、§13、§17、関連する§15.2へ証拠を記録する。
```
