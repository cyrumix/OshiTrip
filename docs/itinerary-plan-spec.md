# 現場詳細「計画」タブ・推し活遠征旅程 仕様書

> 版: 1.1（2026-07-06、Google API低コスト・保存規約準拠方針を反映）
> 対象: Flutter製 iOS／Androidアプリ OshiTrip  
> 関連: [requirements.md](requirements.md) §7.9、[design-spec.md](design-spec.md) §7.3、[ADR-0010](adr/0010-google-maps-platform.md)

## 1. 目的

現場を旅の中心に置き、ライブ・イベントの前後に立ち寄る観光地、飲食店、ホテル、駅、空港等を、移動を含む一つの時系列として管理できるようにする。

一般的な旅行アプリとの差は次のとおりとする。

- 公演日時と会場を旅程の動かせない基準点として扱う
- 既存のチケット、交通、宿泊、Todo・持ち物、思い出と重複入力させない
- 「開場に間に合う」「終演後の最終便に乗れる」を優先して余裕時間を示す
- 計画中の情報はセンシティブ情報としてデフォルト非公開にする
- Google連携が停止・未契約・上限到達でも、手動入力だけで旅程が完成する

## 2. 前提と今回置く仮定

曖昧な部分は次のように定義する。

1. MVPでは現場1件につき既定の旅程を1件作成できる。DBは将来の複数旅程に備えて1対多を許容する。
2. 「宿泊情報から取得」は現在の現場へ登録済みの宿泊を参照する操作を指す。
3. 「交通情報から取得」は現在の現場へ登録済みの往路・復路交通を参照する操作を指す。
4. 取り込んだ交通・宿泊は複製せず、原則として元データを参照する。元データ変更時は計画にも反映する。
5. スポット自体と訪問予定を分離する。同じスポットを別日・別時間に複数回予定できる。
6. 旅程の基準タイムゾーンをIANA形式で保持する。国内MVPの既定値は端末タイムゾーンとし、変更可能にする。
7. Google Places／Maps／RoutesはMVPの主要機能として実装する。ただし機能単位で分離した境界に置き、未設定・障害・上限到達時も手動登録を妨げない。
8. Google APIは検索・最新情報の取得手段であり、Google Maps Contentを永続DBの共有マスタとして再配布する用途には使わない。
9. Google Place IDは規約上保存可能な識別子として永続保存・再利用できる。Google由来の施設名・住所・経路結果は、契約または公式文書で明示的に許可された範囲を超えてキャッシュしない。
10. 共有施設DBと共有概算経路DBへ保存できるのは、ユーザーがGoogle応答から自動転記したものではないユーザー入力、施設提供情報、契約データ、オープンデータ等、保存・再利用権限を説明できるデータに限る。
11. プレミアム判定はAPI呼び出し可否をサーバーで強制する。保存規約・帰属・キャッシュ期限は無料／プレミアムを問わず同じように守る。

## 3. 情報設計

### 3.1 現場詳細のタブ

```text
概要 / Todo・持ち物 / 計画 / チケット / 交通 / 宿泊 / メモ
```

「計画」は現場に紐づく旅程の入口とする。公演会場と公演時間は計画上の固定アンカーとして表示するが、会場情報をスポットとして重複保存しない。

### 3.2 計画タブの表示モード

- `タイムライン`: 日別・時系列の主表示
- `地図`: 登録スポットと選択日の経路を表示
- `候補`: 日時未設定のスポットを一時保管

リリースMVPでは3表示すべてを実装する。開発はタイムラインと候補を先に完成させ、その後に地図を接続する。地図の障害時もGoogle Maps URLによる外部表示とタイムラインを利用可能にする。

## 4. スポット

### 4.1 登録方法

施設名の入力欄は **Google検索と手入力の一体型** にする（入力欄を2つに分けて画面を
複雑にしない, 2026-07-11改訂）。1つの「施設名」欄に文字を入力すると、Google Places が
利用可能な環境では候補が下に出る。

- 候補を選ぶ: 施設名・住所が入力欄へ反映され、Place ID を内部保持する。ユーザーは
  そのまま、または微修正して保存する（Google応答を無確認で永続化しない。永続保存する
  Google由来値は Place ID のみ, §4.3・D-178/D-179）。保存されるのは施設名・住所・Place ID。
- 候補を選ばない: 入力した文字をそのまま手入力の施設名として登録できる。
- Google が未設定・無効・オフライン・上限・障害のとき: 候補は出ず、同じ欄がそのまま
  手入力欄として動く（「そのまま手入力できます」と短く示す）。施設名検索・住所検索の
  どちらの語からも登録できる。

候補表示時は Google の帰属（例: 「候補: Google」）を示し、選択後は「Googleの候補から
選択した」ことを併記する。

### 4.2 手動登録項目

| 項目 | 必須 | 備考 |
|---|---:|---|
| 施設名 | 必須 | 前後空白を除去、空文字不可 |
| カテゴリ | 必須 | §4.4の列挙値 |
| 住所 | 任意 | センシティブ情報として扱う |
| 訪問日 | 任意 | 未設定なら候補リストへ。新規時の既定値は§5.5 |
| 開始時間・終了時間 | 任意 | 終了は開始以後。日跨ぎ可。新規開始時刻の既定値は§5.5 |
| メモ | 任意 | 予約番号等をログへ出さない |
| ユーザー画像 | 任意 | 利用権限を持つ画像のみ |
| リンク | 任意・複数 | §4.5の種別つきURL |
| 前後の余裕時間 | 任意 | 0／15／30／45／60分＋カスタム |

手動登録、編集、削除、並び替えは無料・プレミアムの判定に依存させない。

**緯度・経度は手動入力画面に表示しない**（Phase 2追補）。ユーザーには入力させず、
バリデーション・エラーも出さない。ドメイン／DB の nullable な緯度・経度は将来の
地図・Google 連携用として残すが、この画面からは編集しない。既存スポットの座標は
保存時に保持し（誤って null で上書きしない）、新規手動登録では null とする。

### 4.3 Google Placesから取り込む項目

MVPのPlaces取得は、検索・選択に必要な最小情報だけに限定する。

- Google Place ID（永続保存・再利用可）
- 施設名（検索結果または選択直後の一時表示）
- 住所（検索結果または選択直後の一時表示）
- 必要なGoogle Maps／第三者帰属情報

電話番号、公式サイト、営業時間、評価、レビュー、写真、primary type、座標等はMVPでは取得しない。Field Maskは`id`、`displayName`、`formattedAddress`と、実際の表示に必要な帰属フィールドだけをallowlist化し、`*`を禁止する。

Places API由来の施設名・住所を、API呼び出しの代替を目的としてユーザー横断DBへ恒久保存しない。永続化するGoogle識別子は原則Place IDだけとする。旅程へ永続保存する表示名・住所が必要な場合は、Google応答を自動コピーせず、ユーザーが独立して入力した値を`user_provided`として保存する。Google応答は同一画面・同一操作中の一時状態として扱い、画面終了または定めた短期セッション終了時に破棄する。

共有施設DBは`data_origin`と`rights_basis`を必須にし、Google Places由来コンテンツを投入できないようサーバーで強制する。Place IDは重複候補の照合キーとして共有できるが、名称・住所の権利根拠にはしない。

### 4.5 URL種別

URLは1フィールドへ詰め込まず、種別つきで複数保持する。

- `reference`: 参考URL
- `reservation`: 予約URL
- `google_maps`: Google Maps URL
- `social`: SNS投稿URL
- `ticket`: チケットURL
- `official`: 公式サイトURL
- `other`: その他

`https`を基本とし、外部遷移前にドメインを表示する。危険なスキームは拒否する。

## 5. 旅程とタイムライン

### 5.1 旅程項目

タイムラインは次の項目を同一の流れで表示する。

- スポット訪問
- 公演会場・開場・開演・終演（固定アンカー）
- 登録済み交通から取り込んだ移動
- 登録済み宿泊から取り込んだチェックイン・チェックアウト
- 自由メモ／集合予定
- スポット間の移動区間

### 5.2 日別表示

- 旅程期間の日付ごとにセクションを分ける
- 日付内は開始時刻、手動順、作成時刻の順で決定的に並べる
- 時刻未設定は各日の「時間未定」、日付未設定は「候補」に置く
- ドラッグ並び替えは手動順を変更する。時刻と矛盾した場合は確認を出す
- 日跨ぎ項目は開始日に置き、終了日を併記する
- 端末と旅程のタイムゾーンが違う場合は旅程現地時刻を主表示し、必要に応じて端末時刻も補助表示する

### 5.3 交通・宿泊の取り込み

計画タブに次の操作を置く。

- `登録済みの交通を追加`
- `登録済みの宿泊を追加`

選択画面では既に計画へ追加済みの項目を明示し、重複追加を防ぐ。参照元が削除された場合は旅程項目を勝手に削除せず、「元の交通／宿泊が削除されました」と表示して、手動項目への変換または旅程から削除を選べるようにする。

#### 5.3.1 参照項目の日時は「導出」する（スナップショットしない）

交通・宿泊項目は元データを**複製せず参照**する原則（§2/4）を、名称だけでなく**表示順に関係する日時**にも適用する。方針は「導出（derive）」とし、「同期更新（sync-update）」は採らない。

- 取り込み時に元データの日付・時刻を旅程項目へスナップショットしない（`local_date` を保存しない）。
- タイムラインの表示日・日内の時刻順・融合位置・間に合い判定は、参照元から毎回導出した**実効値**を使う:
  - 交通: 実効開始 = `transports.depart_at`、実効終了 = `transports.arrive_at`、実効表示日 = `depart_at` の暦日。
  - 宿泊: 実効表示日 = `lodgings.checkin_date`。
- したがって元データの出発時刻・チェックイン日などを更新すると、名称だけでなく**表示日・並び順にも即座に反映**される（別日への移動を含む）。参照元が未設定・参照切れなら実効日は未定として候補リストへ置く。

#### 5.3.2 ユーザーによる日時の上書き（override）を明示的にモデル化する

ユーザーが旅程側で独自の日付・時刻を指定したい場合に備え、上書きを**明示的にモデル化**する。

- 旅程項目（`itinerary_entries`）の `local_date` / `start_at` / `end_at` は、交通・宿泊では **null = 参照元に追従、非null = ユーザー上書き**を意味する。
- 上書きがあるフィールドは参照元より優先する（フィールド単位。日付だけ上書き・時刻だけ上書きも可）。
- スポット・メモは参照元を持たないため、これらのフィールドは従来どおり項目自身の値そのもの（null は「未定」）。

この導出・上書きは純粋関数 `effectiveItinerarySchedule`（`itinerary_schedule.dart`）に集約し、タイムライン組み立てと並び替えの同一日検証の両方が同じ規則を使う。

### 5.4 余裕時間

- 各訪問に到着前・出発後の余裕時間を設定できる
- 会場アンカーの推奨既定値は開場30分前とするが、強制しない
- 前項目の終了＋移動時間＋余裕時間が次項目の開始を超える場合、「間に合わない可能性」を文言とアイコンで表示する
- 混雑そのもののリアルタイム予測はMVP対象外。ユーザー設定の余裕時間で代替する

#### 5.4.1 メモ(note)は時間整合の警告対象外（Phase 2追補）

`note`（自由メモ・集合予定）は実際に訪問・移動する予定ではないため、他の予定
との**時間的な整合性判定から完全に除外**する。除外対象:

- 予定時刻の重複警告／並び替え時の時刻矛盾確認
- 移動時間不足・「間に合わない可能性」・前後の余裕時間不足
- 移動区間(leg)の自動的な前後接続対象（端点・隣接判定）
- 新規予定の開始時刻を決める際の「直前予定」（§5.5）

メモ自体はタイムラインへ従来どおり表示し、日付・時刻の入力や形式検証（終了は
開始以後など）は残す。除外するのは「他の予定との時間整合判定」だけ。メモを挟んだ
場合（予定A → メモ → 予定B）は、メモを移動元・移動先として扱わず、A と B の間で
必要な移動時間・余裕を判定する。スポット・交通・宿泊・公演など実予定に対する
警告は維持する。

### 5.5 新規項目の初期値（Phase 2追補）

**訪問日の初期値**（スポット訪問の新規追加時）は、端末の本日ではなく現在操作中の
予定日にする。優先順位:

1. 現在表示・選択しているタイムラインの日付
2. 挿入位置の前後にある予定の日付
3. 現場の開催日
4. 旅程の開始日

いずれかを取得できる場合は本日を初期値に使わない。すべて取得できない場合のみ未設定
（候補リスト）とする。編集時は保存済みの日付を使う。画面を開いただけでは保存せず、
ユーザーは日付を変更できる。日付のみの値とタイムゾーン付き日時を混同しない。

**新規予定の開始時刻の初期値**は、同じ日付のタイムライン上で直前にある「時刻付きの
実予定」の**終了時刻**にする。メモや時刻を持たない項目は読み飛ばし、さらに前の実予定を
探す。直前の実予定に終了時刻が無い（開始のみ）場合は流用せず未設定、前の予定が無い
場合も未設定とする。現在時刻や固定時刻は入れない。初期値設定のみで、保存時に前の予定へ
自動連動はしない。ユーザーが入力済み・編集時の保存値は上書きしない。

導出は純粋関数 `resolveInitialVisitDate` / `resolveInitialStartFromPrevious` に集約する。

## 6. 経路

### 6.1 対応手段

- 徒歩
- 公共交通
- 車
- 自転車
- タクシー
- 飛行機
- その他

Google Routesの明示取得は徒歩・公共交通・車・自転車を中心とし、未対応手段や取得失敗は保存済み概算または手動入力に縮退する。

### 6.2 保存項目

- 出発旅程項目／出発地
- 到着旅程項目／到着地
- 移動手段
- 出発・到着日時
- 所要時間（分）※**手入力欄は廃止**。出発・到着時刻から自動計算した読み取り表示（「所要 約35分」）。保存時に自動計算値を保存（2026-07-11改訂）
- 距離（m）
- 金額（円）※金額欄は残す。**通貨欄は廃止**し日本円前提（JPY固定）。表示は「1,200円」（2026-07-11改訂）
- 乗換・経路概要 ※DB項目は残すが**通常UIには出さない**（入力欄も表示しない）
- 公共交通の路線・停留所・乗換ステップ
- Google Mapsで開くURL ※手動URL入力欄は廃止。経路確認導線（§6.3）から開く
- データソース（手動／Google）
- 取得日時、取得条件、再計算が必要か

移動区間の**通常UI**は、出発地・到着地・移動手段・出発/到着時刻・所要時間（自動計算の
読み取り表示）・距離・金額（円）・「経路を確認（Google Mapsで開く）」導線を中心に表示する。
通貨・経路概要はDB項目としては保持するが、入力欄・一覧表示に出さない（2026-07-11改訂）。

**移動区間の時刻初期値（2026-07-11改訂）**: 出発時刻の初期値は移動元の予定の終了時刻
（無ければ開始時刻）、到着時刻の初期値は移動先の予定の開始時刻（無ければ終了時刻）とする
（端点候補が実効開始/終了時刻を保持し内部決定する）。ユーザーが未入力の間だけ補完し、
入力済みの時刻は尊重する。

移動区間の**出発地・到着地の表示名**は、内部の種別コード（`spot`/`transport`/
`lodging`/`note`）や ID をそのまま出さず、ユーザーが読んで分かる表示名（スポット名／
交通の方向＋手段／宿泊施設名／メモ名、参照切れは「削除済みの…」）を使う。一覧・移動
区間編集の端点候補・出発/到着の選択肢・既存移動区間の表示すべてで同一の生成元を使う
（単一関数 `itineraryTimelineEntryLabel`）。

**移動区間の日付は入力させず、前後予定から内部決定する**（Phase 2追補）。移動区間は
隣接する前後予定を結ぶものなので、編集画面では**時刻だけ**を入力する（別の日付を自由
入力できるUIにはしない）。出発日は原則として出発元（前）の予定日、到着日は原則として
到着先（次）の予定日とする。同日で到着時刻が出発時刻より前になる場合は、日跨ぎとして
到着日を翌日にする。前後予定の日付が取得できない場合は本日を勝手に設定せず、その日時は
未確定（null）とし、所要時間など日付を必要としない内容は保存できるようにするか、日本語で
案内する。既存データの `departure_at` / `arrival_at` との互換性は維持する。日付決定は
純粋関数 `deriveLegTimestamps` に集約する。メモ(note)は端点にしない（§5.4.1）。

永続保存する概算経路は、手動入力、ユーザーが独立して提供した情報、契約データ、オープンデータ等、再利用権限を持つデータを正本とする。Google Routesの応答はライブ表示用の一時データとして明確に分離し、Googleから保存許可を得ていないフィールドを共有概算経路DBへ保存しない。

概算経路で扱う情報は次の最小範囲とする。

- 徒歩距離
- 移動手段
- 路線名
- 乗換概要
- 所要時間
- 金額（取得・保存が許される場合のみ。手動入力可）
- Google Mapsで開くURL（Place ID等からアプリ側で生成）

厳密な時刻表やリアルタイム性は概算経路の目的に含めない。公共交通は旅程作成時点または代表時刻をリクエスト条件に使い、「概算」「取得条件」「最新ではない可能性」を表示する。

運賃は常に任意項目とし、手動入力・修正を許可する。手動値と外部取得値を同じ列で上書きせず、`value_origin`と更新者・更新日時を保持する。

### 6.3 再計算

- スポットの座標、順序、時刻、移動手段が変わったら経路を`要再計算`にする
- 編集やドラッグのたびにAPIを自動実行しない
- 通常表示では、同一の出発・到着・手段に対する保存済みの権利確認済み概算経路を先に表示し、APIを呼ばない
- 経路詳細を開いた時点、または`最新ルートを更新`を押した時点だけGoogle Routesを呼ぶ
- 現仕様では経路検索を全認証ユーザーに開放する（プレミアム区分・課金導線が未整備のため）。Google呼び出しの可否は Edge Function の `ROUTES_REQUIRE_PREMIUM`（既定 `false`）で切り替え、`true` の環境でのみサーバーが entitlement を検証する。既定では認証済みユーザーへ経路検索・保存済み概算経路・手動編集をすべて提供する（D-232）
- 同一ユーザー・同一条件・同時刻の重複リクエストはsingle-flight化する
- Google応答の一時キャッシュは、現行規約で明示的に許可されたフィールド・期間だけに限定し、`expires_at`到達後に削除する。許可が不明な経路内容は永続キャッシュしない
- 公共交通は時刻表変動を考慮し、取得時刻と「最新ではない可能性」を明示する

**経路確認導線（2026-07-11(3)改訂, D-230）**: 移動区間カード（`_LegRow`）と区間編集シートに、
役割を分けた3導線（`RouteLivePanel`）を出す。

- **「経路を確認」（主要・アプリ内）**: 押したときだけ Google Routes をアプリ内で呼び、結果を
  画面内に表示する。代表出発時刻は**区間の出発予定時刻**（`leg.departureAt`、無ければ現在時刻）。
  公共交通では**徒歩合計・路線名・行き先・乗降駅・発着時刻・乗換・合計所要**を乗換タイムライン
  として表示する。結果表示は所要・距離中心で、運賃は通常UIに出さない。端点に Place ID か緯度経度
  があるスポット↔スポットのみ対象。
- **「最新の経路」（当日・移動直前）**: 押した時点の**現在時刻**を出発時刻として、**登録済みの
  出発地**から再取得する。取得結果で保存済み経路を更新するかは確認のうえ、承諾しなければ一時表示
  に留める。
- **「Google Mapsで開く」（補助・外部）**: Google Routes に依存せず、押したときだけ Google Maps の
  経路URLを開く。出発地・到着地に Place ID・緯度経度・**施設名・住所**のいずれかがあれば生成でき、
  Routes が使えない環境でも動く。URLは公式 `dir/?api=1` 形式（`origin`/`destination` は text、
  Place ID は `origin_place_id`/`destination_place_id` 併記、`travelmode` 付与。旧 `place_id:`
  接頭辞は使わない）。起動は external→platformDefault→inApp の順に試し、対応アプリが無くても
  ブラウザで開く。開けないときは理由が分かる案内を出す。

**取得経路の保存（明示変換, D-178/D-180準拠）**: 「この経路を保存」は、Google のライブ結果のうち
乗換ステップ・路線・生レスポンス等の**Googleコンテンツを恒久保存しない**。ユーザーが確認して確定
した**所要時間・距離・Google Maps URL・確認日時（`lastVerifiedAt`）**だけを `user_provided` の
`ItineraryLeg` として保存する（`source`/`fetchedAt`/`cacheKey`/`encodedPolyline` は付けず
`validateItineraryLegPhase1Persistable` を通過。Places の名称・住所と同じ「明示変換」）。APIキー・
秘密情報は保存しない。

**Google リクエストの言語・地域**: Places／Routes／Geocoding へのリクエストは Gateway 層で
`languageCode: ja` / `regionCode: JP`（Routes は加えて `units: METRIC`）を付与し、施設名・住所・
経路案内は日本語表記を優先する。Edge Function 側はサーバー既定（ja/JP/METRIC）でフォールバックする。
Routes Field Mask には発着時刻（`transitDetails.localizedValues.departureTime.time.text`/
`arrivalTime.time.text`）と徒歩所要（`steps.travelMode`/`staticDuration`）を含め、`routes-proxy`
handler が `walkMinutes`（WALKステップ合算）と発着時刻テキストを算出する。

**既存データの言語移行方針（自動上書きしない）**: `languageCode: ja` による日本語優先は
**新規のGoogle取得**に適用される。ja/JP対応より前に保存された既存の施設名・住所
（ユーザーが手入力、またはGoogle候補選択で保存した値）は、アプリが自動で再取得・一括
上書きしない。既存スポット・会場を編集で開いても住所をGoogleへ問い合わせ直さず、保存済みの
値をそのまま表示・保持する（`_SpotEditor.initState` は保存値を読み込むだけで Places を
呼ばない）。既存データを日本語表記へ更新したいときは、**ユーザーの明示操作**——スポット・
会場編集画面でGoogle候補を選び直す「明示変換」（`_selectSuggestion`。永続する Google 由来値は
Place ID のみ, D-178/D-179）——でのみ行う。将来「Googleの情報で更新」導線を設ける場合も、
更新前に確認ダイアログを挟み、ユーザー同意なしに置き換えない。リポジトリ同梱のデモ/seed/
fixture/サンプル/オンボーディング固定データの住所は日本語表記に揃える（英語表記の日本語住所を
残さない。回帰ガード `test/quality/demo_data_japanese_address_test.dart`）。実ユーザーの端末
ローカルデータは自動変更の対象外。

### 6.4 スポット・会場の地図導線・目安表示（2026-07-11追記）

- スポットカード／スポット詳細に「地図で開く」導線を出す。優先度は Place ID（名称＋
  `query_place_id`）→ 手動座標 → 施設名検索。手入力スポットでも施設名（必須）があるので
  常に開ける。URL生成不能か起動失敗かを区別して案内する。
- 会場・公演アンカー（開場/開演/終演）は計画の**予定ではなく「目安」**として表示する。
  予定カード（`AppCard`）ではなく小さめの枠線＋「目安」バッジの補助表示にし、予定と区別
  する（「会場: …」「公演情報: 開演 HH:MM」）。予定として使う場合は「会場を追加」から
  カテゴリ「ライブ・イベント会場」の予定スポット（訪問日=開催日・開始=開場・終了=終演）
  として追加する。会場未登録なら案内し、既に追加済みなら重複確認する。
- 会場入力は「Google検索＋手入力」の一体型UIとし、候補選択で会場名・住所・Place ID を
  保存する（座標は Field Mask 対象外のため保存しない）。Google 無効・未設定でも手入力で
  保存でき、保存した会場の Place ID・住所はスポット同様に地図・経路導線へ使える。

## 7. 画像

### 7.1 ユーザー画像

- スポットごとにユーザー所有画像を登録できる
- 既存のowner分離されたImageStore／Supabase Storage方針を再利用する
- スポットカード、旅程、思い出、アルバムへ利用できる
- 思い出へ追加するときはユーザーの明示操作を必要とする

### 7.2 Google Place写真

API費用と保存・帰属管理を抑えるため、Google Place写真はMVPでは取得しない。スポットカードはユーザー画像またはカテゴリ別プレースホルダーを使う。将来追加する場合も別Phaseで公式規約・SKU・帰属を再確認し、ユーザー所有画像への複製やStorage保存は行わない。

## 8. Google API料金抑制

### 8.1 原則

1. 手動登録を常時利用可能にする
2. APIは画面表示だけで自動実行しない
3. 必要なフィールドだけをField Maskで取得する
4. 検索候補と詳細取得を分ける
5. 経路は明示操作時のみ取得する
6. ユーザー単位・環境単位の上限と停止スイッチを持つ
7. キャッシュ可否はコスト都合で推測せず、Google公式規約で許可されたフィールド・期間をコードとテストへ明示する
8. Google Maps Contentと、自前・ユーザー入力・オープンデータを物理または論理的に分離し、由来不明データを共有キャッシュへ昇格させない

### 8.2 Places検索

- 3文字未満では検索しない
- 400〜500msのdebounceを行い、古いリクエストをキャンセルする
- 検索開始ごとに一意なAutocomplete session tokenを発行し、候補選択のPlace Detailsまで同じtokenを使う
- 候補未選択で画面を閉じたらsessionを終了する
- 現場会場または表示中地図をlocation biasに利用し、結果件数を抑える
- MVPの基本Field MaskはPlace ID、名称、住所と表示に必要な帰属情報だけに限定する
- 電話、Webサイト、営業時間、写真、評価、レビュー、primary type、座標はMVPで取得しない
- 本番でワイルドカードField Maskを禁止する
- Place IDは永続保存・ユーザー横断で再利用できる
- 名称・住所をGoogle応答の共有キャッシュとして保存しない。共有施設DBにはGoogle以外の保存権限を持つデータだけを入れる

**実装状態（2026-07-11）**: 施設名欄の一体型UI（§4.1）から、実 Places ゲートウェイ
（`PlacesGatewayImpl` → places-proxy Edge Function）を接続した。`placesGatewayProvider` は
Places 有効（`googlePlacesAvailable`）かつ Supabase クライアントありのときだけ実 Gateway を
返し、それ以外は `UnavailablePlacesGateway` に縮退する（Web Service用キーはアプリに埋め
込まず places-proxy が保持, ADR-0010 §3）。候補選択で得た施設名・住所は入力欄へ反映し、
ユーザーが確認・保存した値を `user_provided` として永続化する（Place ID のみ Google 由来値
として保持, D-178/D-179）。

### 8.3 Routes

- タイムライン初期表示で全区間を一括計算しない
- 権利確認済みの保存済み概算経路があれば先に表示し、Googleを呼ばない
- 未計算区間でも自動取得せず、ユーザーが「経路を確認」か「最新の経路」を押した場合だけ取得する（現仕様では全認証ユーザーが利用可。将来のプレミアム化は Edge Function の `ROUTES_REQUIRE_PREMIUM=true` 環境でのみ有効、既定 false, D-232）
- Google応答はライブ表示用の一時データとし、保存許可が確認できない経路内容をDBキャッシュしない
- 永続概算経路は手動・ユーザー提供・契約データ・オープンデータを正本にする
- 自動更新は行わず、保存済み概算とライブ結果を明確に区別する
- 無料利用枠を前提にせず、Google Cloud側の日次クォータ、予算通知、アプリ側ユーザー別レート制限を設定する
- 月間利用量が運用閾値へ達した場合、管理者がGoogle自動取得だけを停止できるkill switchを持つ

料金額や無料枠は変更され得るためコードへ固定しない。運用設定と公式料金表をリリース前・四半期ごとに確認する。

### 8.4 Googleコンテンツの保存・共有

- Places APIコンテンツは原則として事前取得・キャッシュ・保存しない。例外としてPlace IDは期限なく保存できる
- Places／Routes由来の緯度・経度を一時キャッシュする場合は、現行の個別規約で認められた最長30日を上限とし、取得元・取得日時・失効日時を保持して自動削除する
- 施設名・住所・経路概要・所要時間・路線・乗換・運賃等をGoogle応答からユーザー横断共有DBへ保存し、将来のAPI呼び出しを代替する設計は採用しない
- Googleから書面による追加許諾または契約上の保存権限を得た場合だけ、feature flag配下で許可フィールド・保存期間・利用範囲を拡張する
- Google由来コンテンツを表示するときは、Google Mapを併用しない表示を含め、公式のGoogle Maps帰属・第三者帰属要件を満たす
- 利用規約と保存可能範囲はリリース前および四半期ごとに公式資料で再確認し、確認日・URL・判断を`docs/decisions.md`へ記録する

## 9. セキュリティ・プライバシー

- Web Service用Google APIキーをアプリへ埋め込まない
- Places Web Service／Routesは認証済みSupabase Edge Function等のサーバー仲介を原則とする
- 地図表示用のクライアントキーはiOS bundle ID等のアプリ制限とAPI制限を付け、環境別に分ける
- Edge Functionは認証、owner、機能フラグ、ユーザー別クォータを検証する
- リクエストログへ住所、座標、旅程本文、予約番号を平文で残さない
- 旅程、住所、座標、予約URL、共有範囲はデフォルト非公開とする
- 共同編集時もowner／editor／viewerをRLSで強制し、UI判定だけに依存しない

## 10. オフライン

- 保存済み旅程、ユーザー入力スポット、メモ、ユーザー画像サムネイル、権利確認済みの概算経路はオフライン閲覧可能にする
- 手動での追加・編集・削除・並び替えはOutboxへ保存し、接続復旧後に同期する
- Google検索とGoogle Routesのライブ取得はオフラインでは行わない
- 地図タイルの完全オフライン保証はしない。地図が使えなくてもタイムライン、ユーザー入力住所、保存済み概算経路を表示する
- 一時表示のみのGoogleコンテンツをオフライン保証の対象にしない

## 11. 共有・共同編集・通知

### 11.1 共有

- 初期状態は本人のみ
- viewerは閲覧のみ、editorは旅程項目を編集可能、ownerだけが共有設定と旅程削除を行える
- 個人メモ、正確な住所、予約URL、位置情報、画像を項目単位で共有対象外にでき、安全側を既定値にする
- 同時編集は更新版による競合検知を行い、黙って上書きしない

共有と共同編集の実装は既存の現場共有基盤完成後とする。

### 11.2 通知

- 出発時刻
- スポット開始時刻
- チェックイン時刻
- 会場へ向かう推奨時刻
- 経路上の余裕不足

OS通知許可は価値説明後に要求し、拒否しても旅程機能は利用可能にする。同一旅程・同一項目・同一種別の通知は冪等にする。

### 12.6 共有施設・共有概算経路

API節約のための共有再利用はGoogleコンテンツのキャッシュではなく、権利確認済みデータで構築する。

- `facility_catalog`: アプリ内施設ID、任意のGoogle Place ID、名称、住所、出典、権利根拠、確認日時、公開状態
- `route_estimates`: 出発施設ID、到着施設ID、手段、代表時刻帯、概算徒歩距離、路線名、乗換概要、概算所要時間、任意運賃、出典、権利根拠、確認日時
- Google Places／Routes由来を示す出典は、Googleとの契約で共有保存が明示許可されている場合以外は登録拒否する
- ユーザー投稿値はowner別の下書きから始め、個人情報・予約情報・正確な行動時刻を除去し、モデレーションまたは信頼度条件を満たしたものだけ共有へ昇格する
- Place IDは照合・重複排除キーとして利用できるが、Place IDが一致するだけで名称・住所をGoogle由来値から補完しない

全子テーブルでownerと親のowner一致をローカル境界とSupabaseトリガーの両方で強制する。削除はplan→spot／entry／leg／linkへ安全にcascadeする。既存transport／lodging削除時は参照切れを検出できる設計にする。

## 13. 状態・エラー

- 初回空状態: 「現場の前後に行きたい場所を追加しましょう」
- API未設定: 手動登録を主CTAとして表示
- API上限: 「自動検索は一時停止中です。手動で登録できます」
- Places候補なし: 入力語を引き継いで手動登録
- 経路取得不可: 出発地・到着地を保持したまま手動入力
- 写真なし: カテゴリ別プレースホルダー
- オフライン: 保存済み内容を表示し、外部取得ボタンを無効化
- 参照元削除: 交通／宿泊を手動項目へ変換または削除
- 同期競合: ローカル／サーバーの差分を示して選択

### 14.4 将来の機能区分候補

現時点では課金制御を実装せず、UseCase／サーバー境界を機能単位で分ける。最終的な無料範囲は利用実績とAPI原価を確認して決定する。

| 常に利用可能にする基礎機能 | プレミアム候補 |
|---|---|
| 手動スポット登録・編集 | Google Places自動入力 |
| 基本旅程・日別タイムライン | Google Routes最新経路の明示更新（※将来候補。現仕様では全ユーザー可, D-232） |
| メモ・複数URL | 複数区間のルート最適化 |
| ユーザー画像 | AI旅程・観光地提案 |
| 既存交通・宿泊の参照 | 高度な画像アルバム |
| 手動経路・余裕時間 | 年間遠征レポート |
| 保存済み旅程の端末内閲覧 | クラウド容量・複数端末・高度なオフライン |
| Google Maps URLで外部表示 | 広告非表示 |

全ユーザーが保存済みの権利確認済み概算経路を閲覧できる。**現仕様では Google Routes のライブ取得（「経路を確認」「最新の経路」）も全認証ユーザーに開放する**（プレミアム区分・課金導線が未整備のため）。Google Routes のライブ取得は将来のプレミアム候補として残すが、制限は Edge Function の `ROUTES_REQUIRE_PREMIUM=true`（既定 `false`）の環境でのみ有効化する。既定ではプレミアム制限をかけず、API 使用量はレート制限・kill switch・timeout・最小 Field Mask・明示操作で制御する（D-232）。旅程作成時に全区間を自動取得する機能は提供しない。

オフライン編集と同期は現在のOshiTrip基盤に含まれるため、既存利用者から安易に取り上げない。クラウド容量や複数端末上限を課金候補として分離する。

## 15. 受入条件

1. Google APIを一切設定しなくても、手動登録だけで旅程を完成できる。
2. 現場会場、交通、宿泊、スポットが同じ日別タイムラインで時系列表示される。
3. 交通・宿泊を重複入力せず参照追加でき、二重追加されない。
4. 時刻未定・日付未定・日跨ぎ・タイムゾーン違いを失わず表示できる。
5. 経路取得失敗時も入力内容を失わず、手動入力へ切り替えられる。
6. Google由来施設情報とライブ経路には必要なGoogle Maps／第三者帰属が表示され、自前・ユーザー入力データと混同しない。
7. オフラインで保存済み旅程を閲覧・編集でき、復旧後に同期される。
8. 別ownerの旅程、スポット、画像、経路をID推測や直接APIで取得・変更できない。
9. API呼び出しがdebounce、session token、最小Field Mask、single-flight、明示操作、クォータで制御される。
10. Dynamic Type、VoiceOver、色覚差、片手操作で主要機能を利用できる。
11. ユーザー横断DBへGoogle由来の施設名・住所・経路結果を無許可で恒久保存せず、Place IDと権利確認済み自前データを再利用する。
12. 非プレミアムは保存済み概算経路を閲覧でき、プレミアムの最新取得が失敗しても概算・手動経路を失わない。
