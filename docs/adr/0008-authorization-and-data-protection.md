# ADR-0008: 認可モデルとデータ保護境界

- 状態: 承認済み
- 日付: 2026-07-01
- 関連要件: §2.4, §7.8, §10.3, §15.2

## 背景 / 課題

推し活情報・座席・チケット・交通・宿泊はセンシティブで、デフォルト非公開（§2.4）。共有は オーナー/編集可/閲覧のみ の3権限、かつチケット画像・予約番号・住所・個人の感想は**項目単位**で共有可否を設定する（§7.8）。共有権限は**サーバー側で検証**しなければならない（§15.2）。これらを継続的に**自動テスト**できる基盤が必要（Phase 0 完了条件）。

## 決定

サーバー（Supabase Postgres）を信頼境界とし、以下で強制する。

1. **所有権**: 全ユーザーデータ行に `owner_id (= auth.uid())` を持たせる。デフォルトで所有者のみアクセス可（RLS `USING (owner_id = auth.uid())`）。
2. **共有**: `genba_shares(genba_id, grantee_id, role, field_grants jsonb)` を介した明示的許可。`role ∈ {owner, editor, viewer}`。RLSポリシーで:
   - viewer: SELECT 可、INSERT/UPDATE/DELETE 不可。
   - editor: 共有対象の編集可。ただしオーナー専用操作（共有設定/削除）は不可。
   - **項目単位**（チケット画像/予約番号/住所/感想）は `field_grants` を参照し、許可されていない項目はビュー/カラムマスキングで非開示。
3. **感想**: ユーザーごとに保持し共同編集しない（§8.4）→ 感想行は `author_id` 基準で本人のみ更新可。
4. **画像**: Supabase Storage を認可付き（`storage.objects` のRLSポリシー）にし、所有者/共有許可者のみ署名URLを取得可能。チケット画像はデフォルト共有対象外（§7.3, §7.8）。
5. **公演マスタ（§10）**: 公演マスタ情報は公開読み取り可。ただしチケット/座席/交通/宿泊/Todo/メモは非公開（別テーブル・別ポリシー）。登録者数は匿名集計のみ、**少人数閾値未満は具体人数を出さない**（Edge Function/ビューで丸め）。作成者編集後の変更履歴を記録し、管理者のみマスタ修正/統合可（`is_admin` 判定）。
6. **監査/削除**: アカウント削除で関連データをカスケード削除（§15.2）。

クライアント側の権限判定はUX最適化のみ。UIを回避してAPIを直接叩いても、RLS/Storageポリシーで拒否される。

## 影響

- スキーマは `supabase/migrations/` に集約し、RLSを**全テーブルで有効化**（`enable row level security`）。
- ログにセンシティブ項目を出さない（`core/logging` の禁止リスト＋テスト、§15.2）。

## 検証方法（自動テスト基盤 — Phase 0 完了条件）

- `supabase/tests/`（pgTAP）で、複数ユーザーの文脈を切替えつつ:
  - 他人の現場/チケット/画像/交通/宿泊にSELECT/UPDATEできないこと。
  - viewer が書込不可、editor がオーナー専用操作不可。
  - 項目単位の共有で非許可項目が読めないこと。
  - 管理者のみ公演マスタを修正/統合できること。
  - 少人数の公演で具体人数が露出しないこと。
- CIで `supabase start` → マイグレーション適用 → pgTAP 実行を必須ジョブにする。
